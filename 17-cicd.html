Pipeline Overview

Every Zeotap service follows a build → test → containerise → push → deploy pipeline. The CI/CD stack uses CircleCI for primary build workflows and GitHub Actions for supplementary tasks (security scanning, dependency auditing).

CircleCI Pipelines

Each service repository contains a .circleci/config.yml. Pipelines are triggered on every push to any branch and run on CircleCI cloud executors.

JVM Services (Scala / Java — SBT)

# .circleci/config.yml (Scala SBT service — e.g., audiences)
jobs:
  build-push-image:
    docker:
      - image: zeotap-docker.jfrog.io/zeotap-maven:sbt
        auth:
          username: $DOCKER_USER
          password: $DOCKER_PASSWORD
    resource_class: large
    steps:
      - checkout
      - setup_remote_docker:
          docker_layer_caching: true
      - restore_cache:
          key: source-{{ checksum "build.sbt" }}

      - run:
          name: Docker Login
          command: docker login --username $DOCKER_USER --password $DOCKER_PASSWORD zeotap-docker.jfrog.io

      - run:
          name: Package JAR (test + coverage)
          command: sbt -Dsbt.log.noformat=true clean test:compile test jacoco dist

      - run:
          name: Build and Push Docker Image
          command: |
            docker build -t audiences .
            docker tag audiences zeotap-docker.jfrog.io/audiences:$CIRCLE_BRANCH
            docker tag audiences zeotap-docker.jfrog.io/audiences:$CIRCLE_BRANCH-$CIRCLE_BUILD_NUM
            docker push zeotap-docker.jfrog.io/audiences:$CIRCLE_BRANCH
            docker push zeotap-docker.jfrog.io/audiences:$CIRCLE_BRANCH-$CIRCLE_BUILD_NUM

      - store_test_results:
          path: target/test-reports

      - save_cache:
          paths: [~/.sbt, ~/.ivy2]
          key: source-{{ checksum "build.sbt" }}

Go Services (e.g., segmentestimator)

# .circleci/config.yml (Go service)
jobs:
  build:
    docker:
      - image: circleci/golang:1.16.0
    environment:
      GAR_LOCATION: europe-west1
      GOOGLE_PROJECT_ID: data-artifacts
      GAR_REPOSITORY: segmentestimator
    working_directory: /go/src/github.com/zeotap/segmentestimator
    steps:
      - checkout
      - restore_cache:
          keys: [v2-pkg-cache]
      - run:
          name: Build
          command: make
      - setup_remote_docker:
          docker_layer_caching: true
      - run:
          name: Build and Push to Google Artifact Registry
          command: |
            docker login -u _json_key_base64 \
              --password ${GOOGLE_DATA_ARTIFACT_GAR_KEY} \
              https://${GAR_LOCATION}-docker.pkg.dev
            docker build -t segmentestimator .

            SHA=$(git rev-parse HEAD)
            SHORTSHA="${SHA:0:7}"
            BRANCH=$(echo "$CIRCLE_BRANCH" | sed 's/\//-/g' | tr '[:upper:]' '[:lower:]')

            if [[ $CIRCLE_BRANCH == *"release"* ]]; then
              TAG="${GAR_LOCATION}-docker.pkg.dev/${GOOGLE_PROJECT_ID}/${GAR_REPOSITORY}/release:${BRANCH}-${SHORTSHA}-${CIRCLE_BUILD_NUM}"
            else
              TAG="${GAR_LOCATION}-docker.pkg.dev/${GOOGLE_PROJECT_ID}/${GAR_REPOSITORY}/${BRANCH}:${SHORTSHA}-${CIRCLE_BUILD_NUM}"
            fi
            docker tag segmentestimator $TAG
            docker push $TAG

      - save_cache:
          key: v2-pkg-cache
          paths: ["/go/pkg"]

workflows:
  version: 2
  build-maybe-publish:
    jobs:
      - build:
          context: Q3-2020

GitHub Actions

With 60+ GitHub Actions workflows across the monorepo, GHA is used for supplementary automation tasks:

Delta Security Scan (delta-scan-action)

# .github/workflows/delta-scan-action.yml
name: Security Scan
on:
  pull_request:
    branches: [main, master, develop]
  push:
    branches: [main, master]

jobs:
  delta-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 0   # full history for diff analysis

      - name: Run Delta Security Scan
        uses: zeotap/delta-scan-action@v2
        with:
          scan_type: "SAST,SCA,SECRETS"
          severity_threshold: "HIGH"
          github_token: ${{ secrets.GITHUB_TOKEN }}
          slack_webhook: ${{ secrets.SECURITY_SLACK_WEBHOOK }}

Release Process

1
Feature Complete — Developer creates PR from feature/xyz → develop. Code review + automated tests must pass.
2
Release Branch — Release manager cuts release/v2.5.0 from develop. CircleCI tags the image as release:v2.5.0-sha-build.
3
QA Deployment — Harness auto-deploys release image to QA. Journey regression suite + backend automation tests run automatically.
4
Manual Gate — QA lead approves in Harness after reviewing test results and regression report.
5
Production Canary — Harness deploys to 10% of production pods. Prometheus monitors error rates and latency for 30 minutes.
6
Full Rollout — If canary is healthy, Harness promotes to 100% of pods. GitHub Release is created automatically by GHA.
