What is the Profile Store?

The Zeotap Profile Store is the central repository for unified customer profiles — a merged, deduplicated, enriched view of every user across all connected data sources. It is built on Delta Lake, giving it ACID transactions, schema evolution, and time-travel (historical versioning) on top of cloud object storage (GCS/S3).

The Profile Store sits at the heart of the Unify pillar. After the Identity Resolution layer assigns a UCID (Universal Customer ID) to each user, the Profile Store collects all attributes, behaviours, and consents belonging to that UCID from:

  • First-party app data — events collected by analytics-sdk, mobile SDKs
  • Data Partner (DPM) data — third-party audience attributes from data partnerships
  • Predictive attributes — model scores generated by the ML platform
  • Post-processed signals — computed attributes (recency, frequency, lifetime value)
Delta Lake — The Profile Store uses the open-source Delta Lake protocol atop GCS buckets, giving it versioned writes, time-travel queries (e.g., VERSION AS OF 42), and concurrent read/write safety without a dedicated data warehouse.

Delta Lake Storage Design

All profile data is stored in Delta Lake tables partitioned by org_id and country. The Delta Lake format provides:

1
ACID Transactions — Concurrent Spark jobs can write simultaneously without corrupting the table, thanks to optimistic concurrency control in the _delta_log.
2
Change Data Feed (CDF) — The Profile Store uses Delta's CDF (delta.enableChangeDataFeed = true) to stream only changed rows downstream, reducing processing cost.
3
Schema Evolution — New attributes from data partners can be added (mergeSchema = true) without rewriting the whole table.
4
Time Travel — Any historical version of the profile table can be queried: spark.read.format("delta").option("versionAsOf", 42) — enabling reproducible audience builds and audits.

Profile Merge Pipeline

The merge pipeline is a multi-stage Spark DAG that combines app events and data partner attributes into a single master profile row per UCID.

sequenceDiagram participant Driver as ProfileMergeDriver participant Config as ConfigEndpoint (HTTP) participant Apps as Apps Delta Table participant DPM as DPM CDF Delta Tables participant Val as ValidationProcessor participant Merge as MultiDPMergeProcessor participant Out as MDM Output Table Driver->>Config: GET profileStoreConfiguration (schema + rules) Config-->>Driver: ProfileStoreConfiguration JSON Driver->>Apps: Read Delta (with recency filter) Apps-->>Driver: AppsInputData DataFrame Driver->>DPM: Read CDF since last checkpoint DPM-->>Driver: DPMergeInputData DataFrames[] Driver->>Val: process(fieldsToIgnore, dataPartnerDetails, config) Val-->>Driver: validated DataFrames Driver->>Merge: process(country, dpDataframes, appsData, outputPath, config) Merge->>Merge: MERGE INTO delta table (upsert by UCID) Merge-->>Out: Write merged profiles to Delta table

Key Scala Classes

object ProfileMergeDriver {
  def main(args: Array[String]): Unit = {
    implicit val spark = SparkSession.builder()
      .config("spark.sql.extensions",
              "io.delta.sql.DeltaSparkSessionExtension")
      .config("spark.sql.catalog.spark_catalog",
              "org.apache.spark.sql.delta.catalog.DeltaCatalog")
      .getOrCreate()

    val profileStoreConfiguration =
      ProfileStoreConfiguration.fromString(
        getHttpResponse(configsEndpoint))

    val dataPartnerDetails =
      parseDatasetInfoFromConfig(dpmDeltaTableCDFConfigs)
        .map { case (path, partnerId, priority) =>
          val df = new DPMergeInputData().getData(path, country)
          DataPartnerDetails(partnerId, priority, df.columns.toList, df)
        }

    val appsData = new AppsInputData()
      .getData(
        DeltaData(appsInputPath, tableVersion = appsVersion),
        getFilterConditionsForAppsMergeData(country, recencyFilter))

    new ValidationProcessor()
      .process(fieldsToIgnoreForValidation,
               dataPartnerDetails, profileStoreConfiguration)

    new MultiDPMergeProcessor()
      .process(country, dataPartnerDetails.map(_.dataframe),
               appsData, outputPath,
               profileStoreConfiguration, inputSchema)
  }
}
class ValidationProcessor {
  // Validates that each data partner's schema matches
  // the expected ProfileStoreConfiguration.
  // Fields in fieldsToIgnoreForValidation are skipped.
  def process(
    fieldsToIgnore: String,
    dataPartnerDetails: List[DataPartnerDetails],
    config: ProfileStoreConfiguration
  ): Unit = {
    val skipFields = fieldsToIgnore.split(",").toSet
    dataPartnerDetails.foreach { partner =>
      val requiredFields = config.requiredFields
        .filterNot(f => skipFields.contains(f.name))
      requiredFields.foreach { field =>
        if (!partner.columns.contains(field.name)) {
          throw new SchemaValidationException(
            s"Partner ${partner.partnerId} missing field: ${field.name}")
        }
      }
    }
  }
}
case class DataPartnerDetails(
  partnerId: String,    // e.g. "rakutenfr.eu"
  priority: Int,        // merge priority (lower = higher priority)
  columns: List[String],
  dataframe: DataFrame
)

// DeltaData wrapper for versioned reads
case class DeltaData(
  path: String,
  tableVersion: Int = -1  // -1 = latest
)

MERGE INTO Strategy

The MultiDPMergeProcessor uses Delta Lake's MERGE INTO (upsert) to atomically update existing profiles or insert new ones:

-- Conceptual SQL generated by MultiDPMergeProcessor
MERGE INTO profile_delta_table AS target
USING incoming_batch AS source
ON target.ucid = source.ucid
WHEN MATCHED THEN
  UPDATE SET
    target.email_sha256   = COALESCE(source.email_sha256, target.email_sha256),
    target.attributes     = MERGE_MAPS(target.attributes, source.attributes),
    target.last_updated   = source.last_updated
WHEN NOT MATCHED THEN
  INSERT (ucid, email_sha256, attributes, created_at, last_updated)
  VALUES (source.ucid, source.email_sha256,
          source.attributes, NOW(), NOW())

Attribute Merge Priority

When multiple data sources provide the same attribute, the priority field on DataPartnerDetails determines which value wins. Lower priority number = higher trust:

PrioritySourceExample Attributes
1 (highest)First-party app eventsemail, phone, purchase history
2Verified data partners (Tier 1)demographics, interests
3Standard data partners (Tier 2)modelled attributes
4 (lowest)Predictive / ML scorespropensity, LTV

Post-Processors

After the main merge, the post-processors module computes derived signals that cannot come directly from raw events or attributes:

1
Recency Scorer — Assigns recency buckets (1/7/30/90/365 days) based on last_seen. Used for freshness-filtered audience segments.
2
Frequency Aggregator — Counts page views, events, and purchases per rolling window (7d, 30d, 90d). Stored as nested app_signals fields.
3
Predictive Attribution — Runs ML model inference (churn, LTV, purchase propensity) and attaches scores to the profile row. Models are versioned and loaded from GCS.
4
DPM-IDU Join Processor — Joins deletion requests from the IDU (Identity Decision Unit) with profile rows to honour GDPR right-to-erasure across the profile table.

Delta Utilities CLI

The deltalake-utilities module provides operational commands for table maintenance:

# Vacuum old snapshots (retain 7 days)
spark-submit --class com.zeotap.delta.Vacuum \
  deltalake-utilities.jar \
  --path gs://bucket/profile-store/4521.eu/delta-table \
  --retentionHours 168

# Optimize (compaction + Z-ordering by ucid)
spark-submit --class com.zeotap.delta.Optimize \
  deltalake-utilities.jar \
  --path gs://bucket/profile-store/4521.eu/delta-table \
  --zorderBy ucid

# Read CDF since version 100
spark.read.format("delta")
  .option("readChangeFeed", "true")
  .option("startingVersion", 100)
  .load("gs://bucket/profile-store/4521.eu/delta-table")

Customer-Facing Profile API

The Profile API is the external REST interface through which client applications read, write, and delete Unified Customer Profiles stored in the Profile Store. It sits in front of the Delta Lake / Aerospike stack and enforces rate limiting, RBAC, and region routing automatically.

Official Docs — This section is aligned with the public API documentation at docs.zeotap.com → API Doc Customer → Profile API. Last synced: March 2025.

POST https://api.zeotap.com/cdp/v1/users/_search

Look up a Unified Customer Profile by one or more identity values. Returns profile attributes, ID graph, consent flags, marketing preferences, and computed AI attributes.

Request Body Parameters

Create or update (upsert) a Unified Customer Profile. Writes profile attributes, marketing preferences, and consent flags atomically for a single identity per call.

Request Body Parameters

ParameterMandatoryTypeDescription

POST https://api.zeotap.com/cdp/v1/users/_delete

Submit a deletion request for one or more user profiles. Triggers the full GDPR erasure pipeline — fan-out across Profile Store Delta, BigTable ID Graph, BigQuery, Audiences DB, and all active Destinations. Supports bulk deletion of up to 400 IDs per call.

Best Practices

  • Call once per session — Cache the response locally; repeated calls count against your quota but return the same data.
  • Use the fetch node — Request only the attributes your app needs to stay under the 1 KB response limit and reduce latency (< 200ms target).
  • Separate API keys per interface — Use distinct keys for web, mobile, and server integrations for granular access control and audit trails.
  • Never call from the browser — Invoke the Profile API only from your back-end personalisation service, not the client side, to prevent API key exposure and XSS vulnerabilities.
  • Skip calls when no ID — Add a guard check: if there is no resolved user ID, skip the API call entirely to preserve quota.
  • Primary IDs only for search — Use IDs designated as Primary IDs in your Zeotap ID Resolution configuration for the most accurate profile match.
