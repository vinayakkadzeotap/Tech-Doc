Audience Management Overview

Audiences are the core output of Zeotap's Unify layer — they define who to reach, before Activate delivers to channels. The Audience Management system consists of four micro-services working in concert:

📋

Audiences API

Play Framework (Scala) service that manages audience CRUD, folder organization, and lifecycle. Central orchestration hub.

📐

Segment Estimator

Go service providing real-time audience size estimates using bitmap indexes and Druid/Qubole/BigQuery sampling.

🔵

Segment Overlap

Computes Jaccard similarity and overlap counts between any two audience segments. Used for reach de-duplication.

⚡

Bulk Segment Creation

Spark-based batch pipeline for creating hundreds of audience segments simultaneously from a single rule manifest.

Segment Estimator Service

The segmentestimator is a Go service (using gin HTTP framework) that answers the critical question: "How many users match this audience definition?" It does so in under 500ms for most queries by combining pre-computed bitmaps with sampled queries.

Bitmap Engine

The estimator uses a compact roaring-style bitmap (backed by []uint64) to represent user sets. Each bit corresponds to a user slot; bitwise AND between two bitmaps gives the intersection count instantly.

// bitmap/bitmap.go
type Bitmap struct {
    values []uint64  // 64 users per uint64 word
    size   int
}

// NewBitmap creates an all-zero bitmap for 'size' users
func NewBitmap(size int) *Bitmap {
    length := (size-1)/64 + 1
    return &Bitmap{
        values: make([]uint64, length),
        size:   size,
    }
}

// AND returns a new bitmap = intersection of a ∩ b
func (a *Bitmap) AND(b *Bitmap) (*Bitmap, error) {
    result := NewBitmap(a.size)
    for i := range a.values {
        result.values[i] = a.values[i] & b.values[i]
    }
    return result, nil
}

// Count returns the number of set bits (Hamming weight)
func (b *Bitmap) Count() int {
    count := 0
    for _, v := range b.values {
        count += bits.OnesCount64(v)
    }
    return count
}

// Extrapolated audience size = Count() / SizePct
func EstimateSize(intersection *Bitmap, sampleRate float64) int64 {
    return int64(float64(intersection.Count()) / sampleRate)
}

Bitmap Lifecycle

1
Ingestion — A Qubole/BigQuery job runs on a 2-hour cadence, pulling a sampled UCID list for each feature extension and writing bitmap binary files to S3/GCS.
2
Loading — On startup and every ESTIMATOR_UPDATE_FREQUENCY seconds, the estimator pulls the latest bitmap files and hydrates in-memory Bitmap objects.
3
Querying — Incoming audience rules are parsed into extension lookups. Bitmaps for matching extensions are AND-ed together, the bit count is extrapolated, and the result is cached.
4
Peer Aggregation — Each estimator replica only holds a subset of the user space (ESTIMATOR_PEER_IP_LIST). The requesting node fans out to all peers and sums the counts.

Bulk Segment Creation

The bulk-segment-creation Spark job enables data teams to create hundreds of audience segments in a single batch from a manifest file — typically used for onboarding a new data partner's taxonomy:

// segment-manifest.json
{
  "org_id": 4521,
  "run_mode": "create",
  "segments": [
    {
      "name": "Luxury Auto Intenders",
      "rules": [
        { "attr": "brand_affinity", "op": "in", "vals": ["BMW","Mercedes","Audi"] }
      ],
      "folder": "Automotive"
    },
    {
      "name": "First-Time Home Buyers",
      "rules": [
        { "attr": "life_stage", "op": "eq", "vals": ["first_home"] }
      ],
      "folder": "Finance"
    }
  ]
}
# Submit bulk segment creation Spark job
spark-submit \
  --class com.zeotap.bulk.SegmentCreationDriver \
  bulk-segment-creation.jar \
  --manifest-path gs://bucket/manifests/q4-segments.json \
  --audiences-api-url http://audiences-svc:9000 \
  --org-id 4521
