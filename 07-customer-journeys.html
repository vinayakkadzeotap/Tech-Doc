What are Customer Journeys?

Customer Journeys in Zeotap — built on the Symphony Workflow Engine — allow marketers to design automated, multi-step customer experiences: from onboarding sequences to re-engagement flows to purchase nurture tracks. A Journey is triggered by an event (e.g., user signup, cart abandonment), then routes the user through conditional logic, time delays, and multi-channel destinations.

The system comprises two components:

  • journeys — The core Play Framework API managing journey CRUD, lifecycle, and execution state.
  • journeys-ai — A FastAPI + Google ADK service where marketers describe a journey in plain language and Gemini 2.5 Flash generates the Symphony Workflow JSON automatically.

Symphony Workflow Engine

Symphony is Zeotap's proprietary workflow orchestration engine. A Symphony Workflow is a directed acyclic graph (DAG) of nodes, where each node is a typed building block. Execution is event-driven: when a user matches the entry condition, a workflow instance is created and the user progresses through the graph asynchronously.

Journey Building Blocks

Block TypePurposeKey Properties
ENTRY_CONDITION Journey entry point. Defines the trigger event and filter predicates that users must match to enter the journey. Required as the first node — a workflow cannot be published without it. event_name, predicates[], re_entry_cooldown_days
CONDITIONAL_SPLIT Binary branching: if a condition is true go to the YES branch, otherwise the NO branch. Evaluates event attributes, profile attributes, ID attributes, user attributes, and NCE data. condition, yes_node_id, no_node_id
MULTIBRANCH Parallel execution: all child branches run simultaneously for the same user. branch_node_ids[]
DELAY Time-based wait. Requires re-enrichment after the delay expires — user data is refreshed before the next action executes. Not a terminal node. duration_value, duration_unit (hours/days/weeks)
DESTINATION Action node: send user data to a configured channel destination (email, push, SMS, ad platform). destination_id, payload_template, channel

Re-entry Logic

After a user completes or exits a journey, re-entry behaviour is configured on the ENTRY_CONDITION node:

Re-entry TypeBehaviourUse Case
No Re-entryUser's UCID is permanently blocked from re-entering the workflowOne-time onboarding sequences
Time-based Re-entryUser is blocked for a set duration (e.g., 10 hours), then eligible to re-enterPeriodic promotions, recurring notifications

Condition Tool — Attribute Types

The Condition Tool (used in CONDITIONAL_SPLIT and entry predicates) can evaluate these attribute categories:

Attribute TypeDescription
Event AttributesProperties of the triggering event (e.g., cart_value, product_id)
Profile AttributesUser profile data from the Profile Store (e.g., user_city, plan_type)
ID AttributesUser identification fields (e.g., email_sha256, ucid)
User AttributesGeneral user properties synced from CRM or SDK events
NCE (Non-Customer Entity)External entity data joined to the primary key of an event or user attribute. NCE fields require a primary key join — either an event attribute or a user/ID attribute

Real-Time Calculated Attributes (RTCAs)

RTCAs are a special category of attributes where the system computes the value based on configured logic, rather than the value being sent directly by the user. The computed value can then be used in conditions just like a regular attribute.

How RTCAs Work

1
Configure the RTCA — Define an event to watch, optional filter conditions, a time window, and an aggregation operation (count, sum, average, etc.).
2
System computes value — As matching events arrive, the system continuously calculates the RTCA value per user. The result is stored on the user's profile.
3
Use in conditions — The computed value is referenced in a CONDITIONAL_SPLIT or entry predicate. E.g., add_to_cart_count > 10.

Example RTCA Configuration

Configuration FieldExample Value
Eventadd_to_cart
Filter Conditioncart_value > 100
Time WindowLast 10 days
OperationCount events matching criteria
Result attributehigh_value_cart_count
RTCA Publish Dependency — RTCAs depend on an external computation service. If that service is unavailable, the RTCA will fail to publish, causing the entire workflow publish to fail. Check the RTCA service status first when debugging publish failures that appear RTCA-related.

Journeys AI — Natural Language to Symphony JSON

The journeys-ai service (FastAPI + Google ADK) lets marketers describe a customer journey in plain English and receive a complete, valid Symphony Workflow JSON. Powered by Gemini 2.5 Flash and a multi-agent ReAct loop.

Agent Definitions

MODEL_NAME = "gemini-2.5-flash"

# Stage 1: Initial creation
workflow_creator_agent = LlmAgent(
    name="workflow_creator_agent",
    model=MODEL_NAME,
    instruction=WORKFLOW_CREATOR_AGENT_INSTRUCTIONS,  # Symphony expert prompt
    tools=[FunctionTool(get_client_catalog)],
    output_key="STATE_CURRENT_WORKFLOW",
)

# Stage 2: Critic — evaluates and suggests improvements
critic_agent = LlmAgent(
    name="critic_agent",
    model=MODEL_NAME,
    instruction=CRITIC_AGENT_INSTRUCTIONS,
    tools=[FunctionTool(verify_generated_workflow)],
    output_key="STATE_CRITICISM",
)

# Stage 2: Refiner — applies critic's suggestions
workflow_refine_agent = LlmAgent(
    name="workflow_refiner",
    model=MODEL_NAME,
    instruction=WORKFLOW_REFINER_AGENT_INSTRUCTION,
    tools=[FunctionTool(get_client_catalog), exit_loop],
    output_key="STATE_CURRENT_WORKFLOW",
)

# Loop: critic + refiner, max 5 iterations
refinement_loop = LoopAgent(
    name="refinement_loop",
    sub_agents=[critic_agent, workflow_refine_agent],
    max_iterations=5,
)

# Root: creator → loop (sequential)
root_agent = SequentialAgent(
    name="root_agent",
    sub_agents=[workflow_creator_agent, refinement_loop],
)

Journey Orchestration & Execution

The audience-orchestration service is responsible for:

1
Trigger Evaluation — Consuming real-time events from Kafka/Pub/Sub and matching users to journey entry conditions based on profile attributes from the Profile Store.
2
State Management — Maintaining per-user journey execution state (current node, entry time, last updated) in PostgreSQL, ensuring exactly-once delivery semantics.
3
Node Execution — For each user-journey state, dispatching the appropriate action: enqueuing destination sends, scheduling delays, evaluating conditional splits.
4
Destination Dispatch — Sending activation payloads to the data-destination service for delivery to channels (email, push, paid media).

Common Publish Failures

1. Data Type Mismatch (Most Common)

The most frequent cause of workflow publish failures. The UI sends values as one type (e.g., list of integers) but the backend API expects another type (e.g., list of strings).

ComponentUI ConfigurationAPI RequestResult
Segment IDs / attribute valuesList of integersExpects list of strings❌ Publish failure / type mismatch error

How to debug: In Groundcover → real-time-distributor, find the publish API request log. Inspect the request payload to identify the mismatched field type.

2. RTCA Publishing Failure

Real-Time Calculated Attributes depend on an external service. If that service is down or slow at publish time, the RTCA fails to publish, cascading into a full workflow publish failure. Check RTCA service health first before investigating the workflow itself.

Execution Pipeline (Full Trace)

Understanding the full pipeline helps pinpoint exactly where a user's journey broke down:

StepStageDescription
1Ingestion & EnrichmentSource-ingestion service maps raw event fields to the Zeotap catalog and adds internal IDs (UCID, org_id)
2Entry Condition EvaluationEnriched event (including any NCE data) is evaluated against all published workflow entry conditions for the org
3Action / DelayIf conditions are met, the Executor triggers the next action (destination send) or schedules a delay
4Re-enrichmentAfter a Delay step expires, user data is refreshed before the next node executes
5CompletionWorkflow finishes. Execution view in Unity Dashboard is updated.
