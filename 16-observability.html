Prometheus Configuration

# prometheus.yml (gcp-prometheus-monitoring/prometheus/prometheus.yml)
global:
  scrape_interval:     1m
  scrape_timeout:      15s
  evaluation_interval: 5m

alerting:
  alertmanagers:
    - static_configs:
        - targets: ["prometheus.zeotap.net:9093"]
      timeout: 30s

rule_files:
  - /etc/prometheus/rules/host_metrics.yml
  - /etc/prometheus/rules/aerospike_metrics.yml
  - /etc/prometheus/rules/scylla_metrics.yml

scrape_configs:
  - job_name: prod-datalake
    gce_sd_configs:
      - project: zeotap-prod-datalake
        zone: europe-west1-b
        refresh_interval: 10m
        port: 9100
      - project: zeotap-prod-datalake
        zone: us-east1-b
        refresh_interval: 10m
        port: 9100
    relabel_configs:
      - source_labels: [__meta_gce_label_team]
        target_label: team
      - source_labels: [__meta_gce_label_role]
        target_label: role
      - source_labels: [__meta_gce_status]
        target_label: instance_status

GCE service discovery automatically picks up all tagged GCE instances across EU and US zones. Labels from GCE instance metadata (team, role) are remapped to Prometheus labels for alert routing.

Alertmanager Configuration

Alertmanager routes alerts to the correct team channel based on label matching:

# alertmanager.yml
global:
  resolve_timeout: 1m
  slack_api_url: https://hooks.slack.com/services/...
  smtp_from: prometheus@zeotap.com
  smtp_smarthost: smtp.sendgrid.net:587
  pagerduty_url: https://events.pagerduty.com/v2/enqueue

route:
  receiver: slack-notifications
  routes:
    - receiver: slack-notifications           # all alerts → Slack
      match:
        notify: devops-engineering
      continue: true

    - receiver: email-notifications           # critical/page → email
      match_re:
        severity: critical|page
      continue: true

    - receiver: pagerduty-notifications       # page → PagerDuty (30m repeat)
      match:
        severity: page
      repeat_interval: 30m

    - receiver: data-engineering-notifications
      match: { team: data-engineering }

    - receiver: backend-engineering-notifications
      match: { team: backend-engineering }

    - receiver: connect-engineering-notifications
      match: { team: connect-engineering }

Receiver Configuration

receivers:

    slack_configs:
      - channel: "#alerts-devops"

Alert Rules

Host / Infrastructure Alerts

# rules/host_metrics.yml
groups:
  - name: common
    rules:
      - alert: node_exporter_down
        expr: up{instance_status="RUNNING",job!="scylla-metrics"} == 0
        for: 10m
        labels:
          severity: critical
          notify: devops-engineering
        annotations:
          summary: "Node exporter down on {{ $labels.instance }}"

      - alert: instance_down
        expr: up{instance_status!="RUNNING"} == 0
        for: 5m
        labels:
          severity: page
        annotations:
          summary: "Instance {{ $labels.instance }} is down"

  - name: data-engineering
    rules:
      - alert: critical_disk_usage
        expr: |
          (1 - (node_filesystem_free_bytes{fstype=~"ext3|ext4|xfs",team="data-engineering"}
                / node_filesystem_size_bytes{fstype=~"ext3|ext4|xfs"})) * 100 > 90
        for: 15m
        labels:
          team: data-engineering
          severity: critical
        annotations:
          summary: "Disk usage > 90% on {{ $labels.instance }}"
          description: "Current value: {{ $value | printf \"%.1f\" }}%"

      - alert: critical_memory_usage
        expr: |
          ((node_memory_MemTotal_bytes{team="data-engineering",role!="as"}
            - node_memory_MemAvailable_bytes)
           / node_memory_MemTotal_bytes) * 100 > 95
        for: 15m
        labels:
          team: data-engineering
          severity: critical
        annotations:
          summary: "Memory usage > 95% on {{ $labels.instance }}"

Application SLO Alerts (GKE services)

  - name: application-slos
    rules:
      - alert: high_error_rate
        expr: |
          rate(http_requests_total{status=~"5.."}[5m])
          / rate(http_requests_total[5m]) > 0.01
        for: 5m
        labels:
          severity: page
          notify: devops-engineering
        annotations:
          summary: "Error rate > 1% on {{ $labels.service }}"

      - alert: high_latency_p99
        expr: |
          histogram_quantile(0.99,
            rate(http_request_duration_seconds_bucket[5m])) > 2
        for: 10m
        labels:
          severity: critical
        annotations:
          summary: "P99 latency > 2s on {{ $labels.service }}"

      - alert: identity_resolution_lag
        expr: |
          kafka_consumer_group_lag{topic="identity-events"} > 100000
        for: 5m
        labels:
          severity: critical
          team: backend-engineering
        annotations:
          summary: "Identity resolution Kafka lag > 100k messages"

Custom Prometheus Exporters

prometheus-bigquery-exporter

Runs SQL queries against BigQuery on a schedule and exposes results as Prometheus gauge metrics — enabling alerting on business-level SLOs:

# BigQuery exporter config
metrics:
  - name: zeotap_daily_profile_count
    query: |
      SELECT COUNT(DISTINCT ucid) as value
      FROM `zeotap-prod.profiles.master_profiles`
      WHERE DATE(last_updated) = CURRENT_DATE()
    interval: 3600s

  - name: zeotap_activation_success_rate
    query: |
      SELECT
        COUNTIF(status = 'SUCCESS') / COUNT(*) as value
      FROM `zeotap-prod.activations.export_events`
      WHERE DATE(exported_at) = CURRENT_DATE()
    interval: 300s

prometheus-query-exporter

A Python-based exporter that runs pre-defined PromQL queries and re-exposes the results as new metrics (useful for pre-computing complex aggregations).

prometheus-query-monitor

Monitors the health of Prometheus itself — checks that key scrape jobs are returning data and that alerting rules are evaluating correctly. Sends Slack alerts if Prometheus appears unhealthy.
