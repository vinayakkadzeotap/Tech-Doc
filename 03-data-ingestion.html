Data Ingestion Pipelines

Zeotap's ingestion layer handles data from first-party web/mobile events, CRM uploads, third-party data providers, and streaming sources — processing billions of events daily via Apache Spark, CDAP, and Apache Beam on GCP Dataflow.

daap · Java/Scala · Maven
Spark · CDAP · Beam/Dataflow
GCS · BigQuery · Pub/Sub

Overview

DAAP (Data as a Platform) is Zeotap's core data platform — a multi-module Java/Scala project that orchestrates all ingestion, processing, and distribution of customer data. It supports two primary modes:

🔄
Batch Ingestion

Scheduled file-based ingestion from SFTP, S3, GCS, CRMs, and third-party data providers. Processed via Spark on Kubernetes.

🌊
Stream Processing

Real-time event processing from Kafka and GCP Pub/Sub topics. Sub-second latency for SDK-generated events.

🔌
3rd-Party Ingestion

Pre-built connectors for 50+ data providers via daap-3P-ingestion. Schema mapping, deduplication, and consent enforcement.

✅
Data Quality

Automated validation via dataquality and data-expectations. Schema drift detection, null rate monitoring, referential integrity.

Batch Ingestion

Batch ingestion runs on a configurable schedule (hourly, daily) using Spark on Kubernetes. Jobs are defined via CDAP pipelines or Oozie workflows.

xml — daap/pom.xml (parent)
<groupId>com.zeotap</groupId>
<artifactId>taube</artifactId>       <!-- internal codename -->
<version>3.0.0-SNAPSHOT</version>
<modules>
  <module>daap-library</module>
  <module>daap-processing</module>
  <module>daap-external</module>
</modules>

Data Quality

ℹ️
Automated Quality Gates

Every batch ingestion job runs through the dataquality module before writing to BigTable/BigQuery. Jobs exceeding configurable error thresholds are quarantined to a dead-letter GCS bucket for manual review.

json — Quality Rule Config
{
  "rules": [
    { "field": "email_sha256", "check": "not_null", "threshold": 0.95 },
    { "field": "email_sha256", "check": "regex", "pattern": "^[a-f0-9]{64}$" },
    { "field": "country",      "check": "enum",  "values": "ISO_3166_ALPHA2" }
  ],
  "onFailure": "quarantine",
  "alertChannel": "slack:#data-quality-alerts"
}

File Ingestion Service Architecture

When a client uploads a file (CSV, JSON, etc.), a chain of services handles the full lifecycle — from raw file on GCS through to final data stores. This is the service-level view from the KT sessions with the data platform team.

UI-Facing Services

Service / RepoRole
data_managerHandles the Unity UI for data uploads: data previews, and saves final field mappings from source fields to Zeotap catalog attributes
canonManages the attribute catalog — the master list of standardised attributes (e.g., country, product name, email). All incoming fields are mapped to canon attributes
source_analyticsPowers the source report in the UI — shows per-file ingestion metadata (file count, record count, status)

Historical ETL Failures — schematic DB

For ingestion failures that occurred in the past (e.g., a file that failed to process yesterday's hourly ETL):

sql — schematic DB: debug ETL failures
-- Step 1: Find failed jobs for an org
SELECT id, org_id, metadata, error, created_at
FROM job_history
WHERE org_id = {org_id}
  AND status = 'FAILED'
ORDER BY created_at DESC
LIMIT 20;

-- The `metadata` column contains the BigQuery job ID
-- The `error` column contains the exact error message

-- Step 2: Use the BQ job ID from `metadata` to inspect
-- the query and full execution details in the BigQuery UI
