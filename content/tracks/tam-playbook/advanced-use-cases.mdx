## 1. Real-Time Personalization Pipelines

Real-time personalization requires sub-second evaluation of user behavior against audience rules, followed by immediate activation to a personalization engine or ad server. This is the most latency-sensitive use case in the CDP and demands a streaming-first architecture.

### How It Works

1. **Event ingestion**: The Zeotap Web or Mobile SDK fires behavioral events (page view, product view, add-to-cart, search) to the streaming ingestion endpoint.
2. **Real-time identity resolution**: The event is matched to a known profile within milliseconds using deterministic identifiers (logged-in user ID, device ID, cookie ID).
3. **Segment evaluation**: The profile is evaluated against all active real-time segments. Segments that use only real-time attributes (current session behavior, last N events) are evaluated on the streaming path.
4. **Activation**: Segment membership changes trigger an outbound webhook or API call to the personalization engine, ad server, or recommendation system.

<Callout type="important" title="Latency Budget">
The total pipeline latency budget is 200ms from event receipt to activation signal. Break this down as: ingestion (20ms) + identity resolution (30ms) + segment evaluation (50ms) + activation API call (100ms). If the customer's personalization engine adds its own processing, the budget is even tighter. Profile the full end-to-end path during implementation.
</Callout>

<Expandable title="Implementation Blueprint" defaultOpen={false}>

**Required connectors**: Zeotap Web SDK, Zeotap Mobile SDK, Server-to-Server Events API, Webhook Destination (for personalization engine)

**Timeline**: 4-6 weeks
- Week 1-2: SDK implementation and event taxonomy design
- Week 3: Real-time identity resolution configuration
- Week 4: Segment creation and webhook activation setup
- Week 5-6: End-to-end latency testing and optimization

**Success metrics**:
- P95 end-to-end latency under 200ms
- Segment evaluation accuracy (validated against batch recomputation) above 99%
- Personalization click-through rate lift of 15-30% vs. non-personalized
- Event throughput capacity matching peak traffic (target: 2x daily peak)

**Key risks**:
- High cardinality segments (e.g., "viewed product X" for millions of products) can slow evaluation
- Identity resolution for anonymous users is probabilistic and less reliable in real time
- Webhook destination availability must be monitored; implement circuit breakers

</Expandable>

<Callout type="tip" title="The Suppression Payoff">
The highest-ROI pattern in cross-channel orchestration is suppression, not activation. When a user converts after an email, immediately suppress them from the paid media retargeting segment. Customers who implement cross-channel suppression typically see 30-40% reduction in media waste with no impact on conversions.
</Callout>

**Success metrics**:
- Cross-channel conversion rate lift of 20-35% vs. single-channel
- Reduction in message fatigue complaints by 50%+
- Paid media spend reduction of 25-40% via suppression
- Customer lifetime value (CLTV) increase of 10-15% over 6 months

### Common Clean Room Use Cases

<Expandable title="Retailer-Brand Collaboration" defaultOpen={true}>

A CPG brand wants to understand which of their website visitors have purchased their products at a major retailer. The brand contributes hashed email addresses from Zeotap; the retailer contributes purchase data with hashed emails. The clean room computes the overlap, and the brand receives aggregate insights (e.g., "35% of your website visitors bought your product in the last 90 days") plus an activatable segment of non-purchasers for targeted media.

**Privacy safeguard**: The brand never sees the retailer's raw data. The retailer never sees the brand's behavioral data. Only aggregate statistics and opaque audience lists leave the clean room.

</Expandable>

<Expandable title="Publisher Audience Enrichment" defaultOpen={false}>

A brand wants to target users on a premium publisher who match their high-value customer profile. The brand shares hashed identifiers and value scores from Zeotap; the publisher shares hashed identifiers and content consumption data. The clean room identifies overlap users and creates publisher-side targeting segments enriched with the brand's value signals.

</Expandable>

<Callout type="info" title="Zeotap's Clean Room Position">
Zeotap integrates with Snowflake Data Clean Rooms, AWS Clean Rooms, and Google Ads Data Hub. Zeotap provides the identity resolution and profile enrichment layer that makes clean room collaboration more effective. The TAM's role is to help the customer define the collaboration terms, select the right clean room technology, and ensure the output segments flow back into Zeotap for activation.
</Callout>

<Expandable title="Implementation Blueprint" defaultOpen={false}>

**Required connectors**: Snowflake Secure Data Share or AWS Clean Rooms connector, Zeotap identity export, partner data ingestion pipeline

**Timeline**: 6-10 weeks (includes legal/commercial negotiation)
- Week 1-2: Identify collaboration partner and define use case scope
- Week 3-4: Legal and data governance review; sign data collaboration agreement
- Week 5-6: Configure clean room environment; define approved query templates
- Week 7-8: Run initial overlap analysis and validate match rates
- Week 9-10: Activate enriched segments through Zeotap destinations

**Success metrics**:
- Match rate between brand and partner data above 20% (varies by industry)
- Enriched audience ROAS improvement of 25-40% vs. non-enriched
- Data governance compliance verified by both parties' legal teams
- Time to insight under 48 hours for approved queries

</Expandable>

### Key Design Decisions

<Expandable title="Merge Rules and Cross-Brand Linking" defaultOpen={true}>

The most critical decision is the merge rule configuration:

- **Strict merge**: Only link profiles across brands when the same verified email address or phone number appears in both. This produces fewer cross-brand links but higher confidence.
- **Probabilistic merge**: Use device IDs, IP addresses, and behavioral signals to link profiles across brands. Higher coverage but risk of false positives.
- **Consent-gated merge**: Only merge profiles across brands when the user has consented to cross-brand data sharing. This is required in GDPR jurisdictions and recommended everywhere.

**TAM recommendation**: Start with strict, consent-gated merge rules. Expand to probabilistic only after the customer has validated the strict merge results and has a clear use case that requires higher coverage.

</Expandable>

<Expandable title="Multi-Region Compliance Patterns" defaultOpen={false}>

For customers operating in multiple regulatory regions (EU, US, APAC):

- **Data residency**: Configure separate Zeotap instances per region where required, or use region-aware processing rules within a single instance
- **Consent mapping**: Map each region's consent framework (GDPR consent, CCPA opt-out, LGPD consent) to Zeotap's consent model
- **Cross-region identity**: Define whether cross-region identity resolution is permitted based on consent. In many cases, EU data cannot be merged with US data without explicit consent for cross-border transfer
- **Audience isolation**: Ensure audiences built on EU data can only be activated to EU-approved destinations

</Expandable>

<Expandable title="Implementation Blueprint" defaultOpen={false}>

**Required connectors**: Per-brand SDKs, per-brand CRM connectors, shared enterprise warehouse connector

**Timeline**: 8-12 weeks
- Week 1-3: Brand-by-brand data audit; map identifiers, consent signals, and data schemas per brand
- Week 4-5: Design merge rules and cross-brand linking strategy; legal review
- Week 6-7: Configure brand-level profile stores and enterprise identity graph
- Week 8-10: Test merge accuracy; validate cross-brand match rates; review edge cases
- Week 11-12: Activate cross-brand use cases (e.g., suppress Brand A customers from Brand B acquisition campaigns)

**Success metrics**:
- Cross-brand match rate of 15-30% (varies by brand overlap)
- False positive rate below 1% (verified via manual audit of sample merges)
- Consent compliance audit passed with zero violations
- Enterprise customer count (deduplicated) accuracy within 5% of manual estimate

</Expandable>

<Expandable title="Integration Design Checklist" defaultOpen={false}>

**Warehouse to CDP**:
- Define the source table(s): typically a customer 360 table with one row per customer
- Map warehouse columns to Zeotap profile attributes (identifiers, traits, computed features)
- Set sync cadence: hourly for active campaigns, daily for steady-state
- Handle schema evolution: what happens when the warehouse schema changes?

**CDP to Warehouse**:
- Export audience membership as a table: user_id, audience_name, entry_timestamp, exit_timestamp
- Export prediction scores from Ada AI back to the warehouse for BI dashboards
- Export event data collected via SDKs back to the warehouse for analytics

**Coordination**:
- Use a shared identifier (typically an internal customer ID) as the join key
- Align on data freshness SLAs: the warehouse export to the CDP and the CDP export to the warehouse should have documented latency expectations
- Monitor sync health with alerting on row count drops, schema mismatches, or sync failures

</Expandable>

### CDP + Marketing Cloud

<Expandable title="Integration Design: Zeotap + Braze / SFMC / Iterable" defaultOpen={false}>

The CDP-to-marketing-cloud integration is the most common activation pattern. Key design considerations:

**Data flow**: Zeotap pushes audience membership and profile attributes to the marketing cloud. The marketing cloud uses these for campaign targeting, message personalization, and triggered sends.

**What to sync**:
- Audience membership (add/remove from lists or segments)
- Key profile attributes for personalization (first name, product affinity, loyalty tier)
- Predictive scores (churn risk, propensity to purchase)
- Do NOT sync raw event streams; the marketing cloud has its own event tracking

**Sync mode**:
- Differential sync (only changes) to minimize API calls and cost
- Full sync on a scheduled cadence (weekly) to correct any drift
- Real-time sync for triggered campaigns (e.g., segment entry triggers a welcome email)

**Identity matching**: Align on the identifier used in both systems. Typically email address or an external customer ID. Ensure the identifier exists in both Zeotap profiles and marketing cloud contact records.

</Expandable>

### CDP + Analytics Platform

<Expandable title="Integration Design: Zeotap + Amplitude / Mixpanel / Adobe Analytics" defaultOpen={false}>

The CDP-to-analytics integration enables enriched behavioral analysis with unified customer profiles.

**Enrichment pattern**: Zeotap sends profile attributes and audience membership to the analytics platform, enabling analysts to segment behavioral data by CDP-derived attributes (e.g., filter funnel analysis by "high-value customer" segment).

**Reverse enrichment**: The analytics platform computes behavioral metrics (session count, feature adoption score) that flow back to Zeotap as profile attributes for audience building.

**Implementation**:
- Use Zeotap's webhook destination or cloud storage export to push enrichment data
- Use the analytics platform's import API or warehouse integration to ingest
- Align user identifiers across both systems
- Set a daily or hourly enrichment cadence depending on analysis requirements

</Expandable>

## 9. Advanced Segmentation Techniques

Standard segment builder operations (filter by attribute, combine with AND/OR) cover 80% of use cases. The remaining 20% require advanced segmentation techniques that unlock significantly more value.

### Behavioral Triggers

Behavioral triggers fire when a user performs a specific action or sequence of actions, rather than matching a static profile attribute.

<ComparisonTable
  headers={["Trigger Type", "Example", "Implementation", "Latency"]}
  rows={[
    ["Single event trigger", "User adds item to cart", "Segment: event 'add_to_cart' occurred in last 30 minutes", "Real-time (streaming path)"],
    ["Event threshold trigger", "User views 5+ products in a session", "Segment: count of 'product_view' events in current session >= 5", "Real-time"],
    ["Absence trigger", "User has not opened app in 7 days", "Segment: last occurrence of 'app_open' event > 7 days ago", "Batch (daily evaluation)"],
    ["Value threshold trigger", "User's cart value exceeds $200", "Segment: sum of 'cart_value' attribute > 200", "Real-time"],
    ["Velocity trigger", "User's purchase frequency increased 2x in last 30 days vs. prior 30 days", "Computed attribute comparing two time windows", "Batch (daily computation)"]
  ]}
  highlightColumn={0}
/>

### Time-Windowed Segments

Time-windowed segments evaluate user behavior within a specific time range, enabling recency-based targeting that static attributes cannot achieve.

<Expandable title="Time Window Patterns" defaultOpen={true}>

- **Rolling window**: "Purchased in the last 30 days" -- the window moves forward each day. Use for ongoing campaigns.
- **Fixed window**: "Purchased between Black Friday and Cyber Monday 2025" -- the window is static. Use for campaign retrospectives and cohort analysis.
- **Relative window comparison**: "Purchase frequency in last 30 days is 2x higher than the 30 days before that" -- compares two rolling windows. Use for trend detection (accelerating engagement, declining activity).
- **Decay window**: Weight recent events more heavily than older ones. "Product affinity score with exponential decay (half-life: 14 days)." Use for recommendation inputs where recency matters more than frequency.

</Expandable>

### Suppression Logic

Suppression segments prevent specific users from receiving messages or being included in paid media audiences. Proper suppression logic is essential for compliance, cost efficiency, and customer experience.

<Expandable title="Suppression Patterns" defaultOpen={false}>

- **Post-conversion suppression**: User purchased the product being advertised. Remove from retargeting audiences within 1 hour of purchase event.
- **Recent communication suppression**: User received an email in the last 48 hours. Suppress from SMS and push to avoid fatigue.
- **Complaint suppression**: User filed a support ticket in the last 7 days. Suppress from all promotional communications.
- **Frequency cap suppression**: User has been exposed to 3+ paid media impressions in the last 7 days. Suppress from paid media to reduce waste.
- **Cross-journey suppression**: User is already in an active retention journey. Suppress from acquisition and cross-sell journeys to avoid conflicting messages.
- **Regulatory suppression**: User has opted out of a specific consent purpose. Suppress from all activations tied to that purpose. This is non-negotiable.

</Expandable>

### Sequential Segments

Sequential segments define an ordered series of events that must occur in sequence, optionally within a time constraint. These are the most powerful and most complex segmentation capability.

<Callout type="tip" title="Sequential Segment Performance">
Sequential segments are computationally expensive because they require ordered event evaluation across potentially millions of user event histories. Limit the lookback window (e.g., last 30 days rather than all time) and limit the number of steps (3-5 is typical) to keep evaluation performant. For real-time sequential segments, consider using the streaming path only for the final step trigger and batch-compute eligibility for earlier steps.
</Callout>
