## ⚙️ Data Ingestion Pipelines

After events pass through the Collect Service, they enter the ingestion pipeline — the backbone that routes, transforms, and delivers data to downstream consumers. This module covers real-time and batch ingestion architectures.

**Topic Design:**

- `events.raw.{orgId}` — Raw events from Collect Service
- `events.validated.{orgId}` — Schema-validated events
- `events.enriched.{orgId}` — Events with identity resolution applied
- `events.dlq.{orgId}` — Dead letter queue for failed events

**Partitioning Strategy:**
- Partitioned by `orgId + userId hash` for ordered processing per user
- 64 partitions per topic (scales with consumer count)
- Retention: 7 days for raw, 30 days for enriched

**Consumer Groups:**
- `id-graph-consumer` — Feeds identity resolution engine
- `profile-consumer` — Updates profile store
- `audience-consumer` — Triggers segment re-evaluation
- `activation-consumer` — Feeds real-time destinations

### Batch Ingestion

**Batch Pipeline Flow:**
1. Customer uploads CSV/JSON to GCS bucket or SFTP
2. File watcher triggers CDAP pipeline
3. Spark job reads file, validates schema, deduplicates records
4. Valid records published to Kafka `events.batch.{orgId}` topic
5. Invalid records written to error report (downloadable by customer)
6. Pipeline metrics emitted to Prometheus

**Batch Processing Features:**
- Configurable scheduling (hourly, daily, weekly, custom cron)
- Incremental processing — only new/changed records
- Large file support — handles files up to 50GB via chunked processing
- Automatic retry with backoff on transient failures

**Dead Letter Queue (DLQ):**
- Failed events routed to `events.dlq.{orgId}` with error metadata
- Error categories: schema_error, consent_missing, rate_limited, processing_error
- DLQ events are inspectable via admin API
- Retry mechanism for transient errors (with exponential backoff)
- Permanent failures archived to cold storage after 30 days

### Backpressure Handling

When downstream consumers can't keep up:

1. **Kafka lag monitoring** — Alert when consumer lag exceeds threshold
2. **Auto-scaling** — Consumer group scales horizontally based on lag
3. **Rate limiting at source** — API Gateway throttles if pipeline is saturated
4. **Priority queues** — Real-time events prioritized over batch backfill
