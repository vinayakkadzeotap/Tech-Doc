## 🏗️ Infrastructure

Zeotap CDP runs on Google Cloud Platform across multiple regions, orchestrated by Kubernetes and provisioned through Infrastructure as Code. ### Node Pools

The production cluster uses specialized node pools to match workload requirements with appropriate machine types, balancing performance and cost.

  ]}
  highlightColumn={0}
/>

**Autoscaling Configuration:**

- **Horizontal Pod Autoscaler (HPA)** -- Scales pod replicas based on CPU, memory, and custom Prometheus metrics (e.g., Kafka consumer lag, request queue depth). Default target: 70% CPU utilization.
- **Cluster Autoscaler** -- Adjusts node count based on pending pods. Scale-up latency target: under 2 minutes. Scale-down delay: 10 minutes of underutilization.
- **Vertical Pod Autoscaler (VPA)** -- Runs in recommendation mode, generating right-sizing suggestions reviewed quarterly. Not used in auto-update mode to avoid unexpected restarts.
- **Scale-to-zero** -- Non-critical batch workloads (report generators, data export jobs) scale to zero during off-peak hours via CronHPA.

<Expandable title="Resource Quotas and Limits" defaultOpen={false}>

Every namespace has resource quotas to prevent runaway resource consumption:

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
spec:
  hard:
    requests.cpu: "100"
    requests.memory: 200Gi
    limits.cpu: "200"
    limits.memory: 400Gi
    pods: "500"
    services: "50"
    persistentvolumeclaims: "20"
```

**Pod-level standards:**
- Every pod MUST define `requests` and `limits` for CPU and memory
- CPU limits set to 2x requests to allow bursting
- Memory limits set to 1.5x requests (OOM-kill is preferred over swap)
- Liveness and readiness probes required for all production deployments
- Pod Disruption Budgets (PDB) ensure minimum availability during node drains

</Expandable>

### Terraform Modules

The `gcp-terraform-modules` repository contains reusable, version-pinned modules for all GCP resources. Every infrastructure change goes through code review, plan approval, and automated apply.

```
gcp-terraform-modules/
├── modules/
│   ├── gke-cluster/         # GKE cluster, node pools, RBAC, network policy
│   ├── bigtable/            # BigTable instances, tables, IAM, app profiles
│   ├── bigquery/            # Datasets, tables, views, scheduled queries
│   ├── kafka/               # Confluent Kafka on GKE, topics, ACLs
│   ├── redis/               # Memorystore Redis instances, failover config
│   ├── networking/          # VPC, subnets, firewall rules, Cloud NAT
│   ├── monitoring/          # Prometheus operator, Grafana, alert rules
│   ├── secrets/             # Secret Manager secrets, IAM bindings
│   ├── dns/                 # Cloud DNS zones, records, health checks
│   └── iam/                 # Service accounts, custom roles, Workload Identity
├── environments/
│   ├── dev/                 # Development environment root module
│   ├── staging/             # Staging environment root module
│   └── production/          # Production environment root module (EU + US)
└── global/                  # Cross-environment resources (DNS, IAM, billing)
```

**State Management:**

| Aspect | Configuration |
|---|---|
| State backend | GCS bucket with versioning and object lifecycle |
| State locking | GCS-based locking to prevent concurrent modifications |
| State structure | Separate state files per environment AND per module |
| Encryption | AES-256 encryption at rest, CMEK for production |
| Access control | IAM-restricted: only CI service account and infra team |

<Callout type="warning" title="Terraform Workflow">
Never run `terraform apply` locally against production. All production changes go through CloudBuild: PR with plan output attached for review, approval by infra team lead, automated apply on merge. Emergency changes require VP Engineering approval and are logged as incidents.
</Callout>

**Module Versioning:**

Modules are versioned using Git tags. Environment configurations pin specific versions:

```hcl
module "gke_cluster" {
  source  = "git::https://github.com/zeotap/gcp-terraform-modules.git//modules/gke-cluster?ref=v3.2.1"

  project_id       = var.project_id
  region           = var.region
  cluster_name     = "zeotap-${var.environment}"
  node_pools       = var.node_pool_config
  enable_istio     = true
  release_channel  = "REGULAR"
}
```

### Multi-Region Deployment

Zeotap operates a multi-region architecture to meet GDPR data residency requirements and provide low-latency access globally. Customer data never crosses region boundaries.

<ArchDiagram
  title="Multi-Region Architecture"
  steps={[
    { label: "Global Load Balancer", detail: "Geo-routing, SSL termination, Cloud Armor WAF", icon: "🌐" },
    { label: "EU Data Plane", detail: "europe-west3 (Frankfurt) — EU customer data", icon: "🇪🇺" },
    { label: "US Data Plane", detail: "us-central1 (Iowa) — US customer data", icon: "🇺🇸" },
    { label: "Shared Control Plane", detail: "Auth, org config, billing, platform admin", icon: "🎛️" },
    { label: "Cross-Region Sync", detail: "Config replication only, never customer data", icon: "🔄" }
  ]}
/>

**Data Plane (per region):**
- Complete, independent deployment of all data-processing services
- Dedicated GKE cluster, BigTable instances, BigQuery datasets, Kafka clusters
- Customer data is ingested, processed, stored, and activated entirely within the region
- No cross-region data transfer for customer PII or behavioral data

**Control Plane (shared):**
- Authentication and user management (Auth0 + platform user service)
- Organization configuration, billing, and license management
- Platform admin tools and internal dashboards
- Replicated across regions for high availability

<Callout type="important" title="Data Residency Guarantee">
When a customer is provisioned in the EU region, all their data -- events, profiles, audiences, identity graph edges -- resides exclusively in `europe-west3`. This is enforced at the infrastructure level: the EU GKE cluster has no network path to US data stores.
</Callout>

**Region Assignment:**
- Each organization is assigned a home region during onboarding
- The region is encoded in the org's configuration and cannot be changed post-provisioning
- API routing uses the org's region to direct requests to the correct data plane
- Global Load Balancer routes based on URL path prefix or org-specific subdomain

<Expandable title="DNS and Traffic Management" defaultOpen={false}>

**DNS Architecture:**
- Cloud DNS manages `zeotap.com` and `*.zeotap.com` zones
- Per-environment subdomains: `dev.zeotap.com`, `staging.zeotap.com`, `app.zeotap.com`
- Health-checked DNS for failover scenarios
- TTL set to 300s for production, 60s for dev/staging

**Traffic Flow (external request):**
1. Client resolves `app.zeotap.com` via Cloud DNS
2. Request hits Global HTTPS Load Balancer (Anycast IP)
3. Cloud Armor evaluates WAF rules (OWASP Top 10, rate limiting, geo-blocking)
4. SSL terminated at the load balancer (managed certificates, auto-renewal)
5. Request routed to the appropriate regional backend service
6. Istio sidecar handles mTLS for all intra-cluster communication

**Firewall Rules:**
- Default deny all ingress and egress
- Explicit allow list for known CIDR ranges (GCP services, partner integrations)
- Kubernetes Network Policies (Calico) enforce pod-to-pod communication rules
- All firewall changes require code review via Terraform

</Expandable>

**Secret Categories:**

| Category | Examples | Rotation Cycle |
|---|---|---|
| Database credentials | BigTable, Redis, PostgreSQL passwords | 90 days (automated) |
| API keys | Auth0 management API, destination partner keys | 180 days (manual review) |
| Encryption keys | Data-at-rest encryption, token signing | Annual (CMEK via Cloud KMS) |
| TLS certificates | Internal mTLS, webhook endpoint certs | Auto-renewed via cert-manager |
| Service account keys | GCP SA keys for external integrations | 90 days (automated, prefer Workload Identity) |

<Callout type="tip" title="Workload Identity over SA Keys">
Prefer GKE Workload Identity over exported service account keys. Workload Identity maps Kubernetes service accounts directly to GCP service accounts without managing key files. This eliminates key rotation overhead and reduces the attack surface significantly.
</Callout>

- Budget alerts at 80% and 100% of quarterly allocation
- FinOps review in sprint retrospectives for teams exceeding budget

<Expandable title="Capacity Planning Process" defaultOpen={false}>

Quarterly capacity planning ensures infrastructure scales ahead of demand:

1. **Traffic forecasting** -- Analyze growth trends, factor in new customer onboarding pipeline
2. **Load test validation** -- Run load tests at 2x projected peak to identify bottlenecks
3. **Resource modeling** -- Map projected traffic to CPU, memory, storage, and network requirements
4. **CUD evaluation** -- Review committed use discount opportunities based on new baseline
5. **Budget proposal** -- Submit infrastructure budget for next quarter with cost-benefit justification
6. **Provisioning** -- Terraform changes to adjust node pool limits, storage quotas, BigTable node count

The infra team maintains a capacity dashboard in Grafana showing current utilization vs. provisioned capacity for all critical resources with 30-day trend projections.

</Expandable>
