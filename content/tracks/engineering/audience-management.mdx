### Segment Rule DSL

Segment rules are defined in a JSON-based DSL:

```json
{
  "operator": "AND",
  "conditions": [
    { "field": "attributes.country", "op": "eq", "value": "DE" },
    { "field": "behaviors.total_purchases", "op": "gte", "value": 5 },
    { "field": "events.cart_abandon", "op": "occurred_within", "value": "7d" }
  ]
}
```

**Supported Operators:**
- Comparison: `eq`, `neq`, `gt`, `gte`, `lt`, `lte`
- String: `contains`, `starts_with`, `ends_with`, `matches` (regex)
- Set: `in`, `not_in`
- Temporal: `occurred_within`, `not_occurred_within`, `between_dates`
- Existence: `exists`, `not_exists`
- Logical: `AND`, `OR`, `NOT` with unlimited nesting

### Evaluation Engine

**Real-Time Evaluation:**
1. Profile change event arrives (new attribute, new event, identity merge)
2. Engine identifies which segments reference the changed fields
3. Only affected segments are re-evaluated (not all segments)
4. Membership changes (enter/exit) published to `audience.changes.{orgId}` topic

**Batch Evaluation:**
- Scheduled full recomputation for complex segments (ML-based, time-decay)
- Runs on Spark for parallelism across millions of profiles
- Reconciles any drift between real-time and batch

### Audience Size Estimation (Bitmap Engine)

The `segmentestimator` service answers "How many users match?" in **&lt;500ms** using a **roaring-style bitmap engine**:

**How It Works:**
1. Parse audience rules into extension lookups
2. Check in-process LRU cache (TTL: 7200s / 2 hours)
3. If cache miss: load bitmap files from S3/GCS
4. Bitwise AND across all rule bitmaps (`[]uint64`, 64 users per word)
5. Popcount (Hamming weight) x inverse sample rate = estimated size
6. Fan out to peer replicas and aggregate counts
7. Cache result for future queries

**Bitmap Data Structure (Go):**
- Each feature extension is a `[]uint64` bitmap — 64 users per uint64 word
- Bitwise AND between two bitmaps = instant intersection
- Sample rate configurable per region (e.g., 0.1% for India)
- Bitmaps refreshed every 2 hours via Qubole/BigQuery ingestion jobs

**Configuration:**
- `ESTIMATOR_DATA_REFRESH=enabled` — auto-refresh bitmaps
- `ESTIMATOR_UPDATE_FREQUENCY=7200` — refresh every 2 hours
- `ESTIMATOR_PEER_IP_LIST=10.0.0.1:3000,10.0.0.2:3000` — replica peers for fan-out
- `ESTIMATOR_FETCHER=bigquery` — data source (qubole or bigquery)

### Audience Overlap Analysis (segment-overlap)

The `segment-overlap` service computes set comparisons between audiences:

- Takes two audience segment UCIDs
- Calculates intersection via bitmap AND
- Returns: `overlap_count` (|A ∩ B|), `jaccard_similarity` (|A ∩ B| / |A ∪ B|), `a_only_count`, `b_only_count`
- Use cases: reach planning, suppression lists, lookalike seeding

### Audience Export

Audiences can be exported in multiple formats:
- **Real-time stream** — Kafka topic with enter/exit events
- **Batch file** — CSV/JSON to GCS on schedule
- **API** — Query membership via REST API
- **Destination push** — Automatic sync to activated destinations

### Real-Time Calculated Attributes (RT CAs)

**RT CAs** enable dynamic profile enrichment computed in real-time as events arrive:

- Compute derived attributes on-the-fly (e.g., "cart value in current session", "pages viewed today")
- Used by the **Interact SDK** for client-side personalization at millisecond latency
- Powers real-time decisions without waiting for batch computation
