### Logs (Structured Logging)

- **Format:** JSON structured logs with standardized fields
- **Fields:** `timestamp`, `level`, `service`, `traceId`, `orgId`, `message`, `error`
- **Storage:** GCP Cloud Logging with 30-day retention
- **Long-term:** Exported to BigQuery for analysis beyond 30 days
- **PII:** Never log PII — use hashed identifiers only

### Traces (OpenTelemetry)

Distributed tracing tracks requests across microservices:

- **OpenTelemetry SDK** instrumented in every service
- **Trace context** propagated via HTTP headers (W3C Trace Context)
- **Spans** record timing for each operation (DB query, API call, Kafka publish)
- **Sampling:** Head-based sampling at 1% for production, 100% for staging
- **Visualization:** Cloud Trace for trace viewing and analysis

**Example Trace:**
```
Browser → API Gateway (5ms) → Collect Service (12ms) → Kafka Publish (3ms)
                                                     → Schema Validation (8ms)
```

### Alerting

**Alert Routing:**
1. Prometheus Alertmanager evaluates alert rules
2. Critical alerts → PagerDuty → on-call engineer (immediate)
3. Warning alerts → Slack #alerts channel → team review
4. Info alerts → Grafana dashboard annotation

**On-Call Rotation:**
- Weekly rotation per team (Collect, Identity, Activation, Platform)
- Primary + secondary on-call for coverage
- Runbooks linked in every alert for fast resolution
- Post-incident review for every P1/P2 alert

### Key Dashboards

Every team maintains these Grafana dashboards:
- **Service Overview** — RED metrics, error breakdown, latency distribution
- **Infrastructure** — Pod count, CPU/memory, node health
- **Business Metrics** — Events ingested, profiles resolved, syncs completed
- **SLO Tracker** — Error budget consumption over time

<Callout type="tip" title="Dashboard Hygiene">
A dashboard with 50 panels is useless in an incident. Keep dashboards focused: 6-8 panels per dashboard, clear titles, and consistent time ranges. Create separate dashboards for investigation vs. monitoring.
</Callout>
