## 📊 Observability

Observability is how Zeotap engineers understand what is happening inside a distributed system of hundreds of microservices. This module covers the three pillars -- metrics, logs, and traces -- along with SLOs, alerting, incident response, and on-call practices.

**Custom Business Metrics:**

Beyond RED, services expose domain-specific metrics:

- `events_ingested_total` -- Events processed per source per org
- `identity_resolutions_total` -- ID graph stitching operations per second
- `audience_evaluations_duration_seconds` -- Segment evaluation latency
- `destination_sync_success_total` -- Successful audience syncs per destination
- `kafka_consumer_lag` -- Consumer group lag per topic partition

<Expandable title="Instrumentation Standards" defaultOpen={false}>

All services follow these instrumentation standards:

**Java/Spring Services:**
- Micrometer library with Prometheus registry
- Spring Boot Actuator exposes `/actuator/prometheus`
- `@Timed` annotation on controller methods for automatic histogram metrics
- Custom `MeterBinder` implementations for business metrics

**Go Services:**
- `prometheus/client_golang` library
- HTTP middleware auto-records request duration and status
- Custom collectors for identity graph operations and BigTable latency

**Scala/Spark Jobs:**
- Spark metrics sink configured to push to Prometheus Pushgateway
- Job duration, records processed, and error counts per batch run

**Naming Conventions:**
- Snake_case metric names: `http_request_duration_seconds`
- Labels for dimensionality: `service`, `method`, `status`, `org_id`
- Histograms for latency (not summaries) to allow aggregation
- Counter suffix `_total`, histogram suffix `_seconds` or `_bytes`

</Expandable>

### Structured Logging (GCP Cloud Logging)

Logs provide the detailed context that metrics cannot. Every service writes structured JSON logs with standardized fields to enable consistent querying and correlation.

**Standard Log Fields:**

```json
{
  "timestamp": "2024-01-15T10:30:45.123Z",
  "severity": "ERROR",
  "service": "audience-engine",
  "version": "v2.14.3",
  "traceId": "4bf92f3577b34da6a3ce929d0e0e4736",
  "spanId": "00f067aa0ba902b7",
  "orgId": "org_12345",
  "userId": "user_abc",
  "message": "Failed to evaluate audience segment",
  "error": {
    "type": "BigTableReadException",
    "message": "Deadline exceeded after 5000ms",
    "stackTrace": "..."
  },
  "context": {
    "audienceId": "aud_xyz",
    "segmentSize": 150000
  }
}
```

**Logging Rules:**

- **Never log PII** -- Use hashed identifiers (`sha256(email)`) instead of raw values
- **Always include traceId** -- Links logs to distributed traces for request correlation
- **Always include orgId** -- Enables tenant-scoped log queries during debugging
- **Use appropriate severity** -- DEBUG for development, INFO for operational events, WARN for recoverable issues, ERROR for failures requiring attention
- **Structured context** -- Use nested JSON objects for request-specific context, not string interpolation

**Log Storage and Routing:**

| Destination | Retention | Use Case |
|---|---|---|
| Cloud Logging | 30 days | Real-time querying, recent incident investigation |
| BigQuery (log sink) | 1 year | Long-term analysis, compliance auditing, trend analysis |
| Error Reporting | 30 days | Automated error grouping and notification |

<Callout type="tip" title="Log-Based Metrics">
GCP Cloud Logging can create metrics from log entries. Use log-based metrics for signals that are expensive to instrument in code, such as counting specific error types or tracking configuration change events.
</Callout>

### Distributed Tracing (OpenTelemetry)

Distributed tracing tracks a single request as it flows across multiple microservices, revealing latency bottlenecks and dependency issues that metrics alone cannot surface.

<ArchDiagram
  title="Trace Propagation Flow"
  steps={[
    { label: "SDK / Browser", detail: "Generates root trace ID", icon: "📱" },
    { label: "API Gateway", detail: "Adds gateway span, propagates W3C headers", icon: "🛡️" },
    { label: "Service A", detail: "Creates child span, calls Service B + Kafka", icon: "⚙️" },
    { label: "Service B", detail: "Creates child span, queries BigTable", icon: "🗄️" },
    { label: "Cloud Trace", detail: "Aggregates spans into full trace view", icon: "🔍" }
  ]}
/>

**OpenTelemetry Configuration:**

- **SDK:** OpenTelemetry SDK instrumented in every service (Java, Go, Python, TypeScript)
- **Propagation:** W3C Trace Context headers (`traceparent`, `tracestate`) for cross-service correlation
- **Auto-instrumentation:** HTTP clients, gRPC, JDBC, Redis, and Kafka are auto-instrumented
- **Custom spans:** Engineers add manual spans for business-critical operations (e.g., audience evaluation, identity resolution)
- **Sampling:** Head-based sampling at 1% for production traffic, 100% for staging and error traces
- **Export:** Spans exported to GCP Cloud Trace via the OpenTelemetry Collector

<Expandable title="Trace Example: Event Ingestion" defaultOpen={false}>

A single event ingestion request generates spans across multiple services:

```
[Root] POST /collect/event (API Gateway) ................ 45ms
  ├── [Auth] Token validation ........................... 3ms
  ├── [Collect] Schema validation ....................... 8ms
  ├── [Collect] Consent check .......................... 2ms
  ├── [Kafka] Publish to events.raw.org_123 ............. 5ms
  ├── [ID Graph] Identity resolution ................... 18ms
  │     ├── [BigTable] Read identity edges .............. 7ms
  │     ├── [Graph] Component merge ..................... 4ms
  │     └── [BigTable] Write updated edges .............. 6ms
  └── [Profile] Upsert profile attributes ............... 9ms
        ├── [BigTable] Read current profile ............. 4ms
        └── [BigTable] Write updated profile ............ 5ms
```

This trace immediately shows that identity resolution (18ms) is the largest contributor to overall latency, with BigTable reads as the bottleneck.

</Expandable>

### SLOs and SLIs

Service Level Objectives define the reliability targets for critical services. Each SLO is measured by a Service Level Indicator and has an associated error budget.

  ]}
  highlightColumn={2}

<Callout type="info" title="Error Budget Policy">
When a service exhausts its monthly error budget, the owning team pauses feature work and focuses exclusively on reliability improvements until the budget is restored. This policy is enforced by engineering leadership and tracked in the weekly SRE review.
</Callout>

**SLO Burn Rate Alerts:**

Rather than alerting on instantaneous threshold breaches, Zeotap uses multi-window burn rate alerting:

- **Fast burn (2% budget consumed in 1 hour)** -- Page on-call immediately (P1)
- **Slow burn (5% budget consumed in 6 hours)** -- Alert to Slack, review within shift (P2)
- **Informational (10% budget consumed in 3 days)** -- Weekly SRE review agenda item (P3)

### PagerDuty Integration and Alerting

Alerts flow from Prometheus Alertmanager through a structured routing pipeline to ensure the right person is notified at the right time.

**Alert Severity Levels:**

| Severity | Response Time | Escalation | Example |
|---|---|---|---|
| P1 (Critical) | Acknowledge in 5 min | Auto-escalate to secondary after 10min, manager after 30min | Platform-wide outage, data loss risk |
| P2 (High) | Acknowledge in 15 min | Escalate to secondary after 30min | Single service degraded, SLO burn rate high |
| P3 (Medium) | Review within shift | No automatic escalation | Elevated error rate, non-critical service down |
| P4 (Low) | Review next business day | None | Resource utilization warning, non-urgent maintenance |

**Incident Lifecycle:**

1. **Detection** -- Alert fires, PagerDuty pages on-call engineer
2. **Acknowledge** -- On-call acknowledges within SLA, opens Slack incident channel
3. **Triage** -- Assess severity and impact using dashboards and traces
4. **Mitigate** -- Apply fix or rollback using runbook procedures
5. **Resolve** -- Confirm service recovery via metrics and customer reports
6. **Post-mortem** -- Blameless review within 48 hours for P1/P2 incidents, action items tracked in Jira

<Expandable title="Runbook Structure" defaultOpen={false}>

Every alert links to a runbook with the following structure:

1. **Alert description** -- What this alert means, what service is affected
2. **Impact assessment** -- How to determine customer impact and severity
3. **Dashboard links** -- Direct links to relevant Grafana dashboards
4. **Investigation steps** -- Ordered checklist of diagnostic queries and commands
5. **Mitigation options** -- Possible fixes ranked by speed and risk (rollback, restart, scale, config change)
6. **Escalation criteria** -- When to escalate to secondary on-call or engineering manager
7. **Communication template** -- Pre-drafted customer communication for statuspage

Runbooks are stored in the `runbooks/` directory of the DevOps repo and reviewed quarterly for accuracy.

</Expandable>

**On-Call Practices:**

- **Weekly rotation** per domain team (Collect, Identity, Activation, Platform)
- **Primary + secondary** on-call for coverage and escalation
- **Handoff meeting** at rotation change with open issues and context
- **On-call compensation** -- Additional pay for after-hours pages
- **Toil tracking** -- Repetitive incident responses are logged as toil and prioritized for automation

<Callout type="warning" title="Dashboard Hygiene">
A dashboard with 50 panels is useless during an incident. Keep dashboards focused: 6-8 panels per dashboard, clear titles, consistent time ranges, and a logical reading order (top-to-bottom, left-to-right). Create separate dashboards for monitoring (glanceable) vs. investigation (detailed drill-down).
</Callout>
