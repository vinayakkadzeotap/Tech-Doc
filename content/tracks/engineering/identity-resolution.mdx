## 🔗 Identity Resolution

Identity resolution is Zeotap's core differentiator. It stitches fragmented identifiers across devices, channels, and touchpoints into a single Unified Customer Identity (UCID). This module covers the graph-based approach, matching algorithms, and storage model.

### Graph Data Model

The identity graph is modeled as a property graph:

**Nodes (Identifiers):**
- Each identifier is a node: `email:user@example.com`, `cookie:abc123`, `device:xyz789`
- Nodes have a type (email, phone, cookie, device, crm_id) and value
- Nodes carry metadata: first_seen, last_seen, source

**Edges (Observations):**
- An edge connects two identifiers that were observed together
- Edges carry: timestamp, source, confidence score, observation count
- Example: Cookie `abc123` was seen with email `user@example.com` during a login event

**UCID (Cluster):**
- Connected components in the graph form a cluster = one customer
- Each cluster gets a stable UCID (UUID)
- The UCID is the primary key for the Profile Store

### Matching Strategies

<ComparisonTable
  headers={["Strategy", "Method", "Accuracy", "Coverage"]}
  rows={[
    ["Deterministic", "Exact match on known IDs (email, phone)", "99.7%", "Lower (needs known IDs)"],
    ["Probabilistic", "Statistical matching (device signals, IP, timing)", "85-95%", "Higher (works with anonymous)"],
    ["Transitive", "A=B and B=C implies A=C", "Varies", "Extends reach of deterministic"]
  ]}
  highlightColumn={2}
/>

**Deterministic Matching:**
- Email-to-email exact match (normalized, lowercased, hashed)
- Phone-to-phone match (E.164 normalized)
- Login ID match across platforms
- CRM ID match from batch imports

**Probabilistic Matching:**
- Device fingerprinting signals (screen resolution, timezone, language, OS)
- IP address clustering (same household)
- Behavioral patterns (similar browsing patterns from same location)
- Configurable confidence thresholds per org

1. For each identifier in the event, look up existing graph node
2. If identifiers belong to different clusters, evaluate merge
3. Merge criteria: deterministic link (always merge) or probabilistic link above threshold
4. On merge: smaller cluster absorbed into larger, UCID of larger cluster wins
5. Emit merge event to downstream consumers (Profile Store, Audience Engine)

**Split Detection:**
- Monitor for clusters growing suspiciously large (>1000 identifiers)
- Flag shared devices, generic emails, bot traffic
- Configurable exclusion lists (e.g., `noreply@`, `info@`)
- Manual split capability via admin API

### Storage Architecture

**BigTable for Graph Storage:**

| Row Key | Column Family | Columns |
|---------|--------------|---------|
| `{orgId}#{identifierType}#{identifierValue}` | `edges` | Connected identifiers + metadata |
| `{orgId}#ucid#{ucidValue}` | `members` | All identifiers in cluster |
| `{orgId}#ucid#{ucidValue}` | `meta` | Cluster size, created_at, updated_at |

- BigTable chosen for low-latency random reads ({'<'}10ms p99)
- Row key design enables efficient lookups by identifier or UCID
- Automatic compaction and TTL for stale edges (configurable, default 90 days)

**Why Go?**

The identity resolution service (`id-graph-lib-v2-go`) is written in Go because:
- High concurrency with goroutines (millions of concurrent lookups)
- Low memory footprint per connection
- Sub-millisecond GC pauses
- Easy deployment as a single binary on GKE

<Callout type="info" title="Performance at Scale">
The Go-based identity service handles 500K identity resolution requests per second with {'<'}50ms p99 latency. It runs as a horizontally scaled deployment on GKE with 20-50 pods depending on load.
</Callout>

### ID+ Universal Identity Solution

**ID+** is Zeotap's standalone universal identity product, also used within the CDP:

- Creates a first-party ID+ cookie unique per browser/website/device
- Identifies users at a non-PII individual level
- Attaches deterministic identities when users log in
- Does NOT rely on third-party cookies, MAIDs, or platform identifiers
- **~50 billion IDs** tracked and correlated globally
- **400M+** ID-verified/self-declared linkages
- Available across North America, Latin America, Europe, and India
- Joined the **Prebid ecosystem** for publisher-side identity resolution

### Architecture Evolution

The identity graph has undergone significant migrations:

| Era | Storage | Latency | Key Change |
|-----|---------|---------|------------|
| Pre-2020 | JanusGraph on ScyllaDB (AWS) | ~600ms | World's largest native JanusGraph database |
| 2020-2022 | Direct ScyllaDB via CQL | ~30ms | Removed JanusGraph layer, used CQL directly |
| 2024+ | Google BigTable | {'<'}10ms | Cloud-native, reduced operational footprint by 20% |

### Privacy Considerations

- All PII identifiers are hashed (SHA-256) before storage
- Graph edges respect consent — no linking without valid consent
- Right-to-erasure deletes all nodes and edges for a given identifier
- Data residency enforced — EU graph data stays in EU region
