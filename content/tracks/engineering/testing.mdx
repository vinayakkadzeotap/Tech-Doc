**Best Practices:**
- Test behavior, not implementation
- Use dependency injection for testability
- Mock external dependencies (databases, APIs, Kafka)
- Avoid flaky tests — quarantine immediately if found
- Run on every PR as a merge gate

### Integration Testing

Integration tests verify a service works with its real dependencies:

- **Database tests** — Service + BigTable emulator or test BigQuery dataset
- **Kafka tests** — Service + embedded Kafka (Testcontainers)
- **API tests** — Service + HTTP client against running instance
- **Cache tests** — Service + Redis via Testcontainers

**Test Environment:**
- Docker Compose for local integration testing
- Testcontainers for ephemeral dependency containers
- Shared staging environment for cross-service integration
- Test data factories for consistent, reproducible test data

### Contract Testing (Pact)

In a microservices architecture, API contracts between services are critical:

**How Pact Works:**

1. **Consumer** writes a test defining expected API behavior
2. Pact generates a **contract file** from the consumer test
3. **Provider** verifies the contract against its actual implementation
4. Contract stored in **Pact Broker** for versioned tracking
5. Breaking changes detected before deployment

**Benefits:**
- Catch breaking API changes without running full E2E suite
- Each team tests independently against contracts
- Faster feedback than end-to-end testing
- Contracts serve as living API documentation

### End-to-End Testing

**Frontend E2E (Cypress):**
- Critical user flows: login, create audience, set up destination, view reports
- Run against staging environment on every deploy
- Visual regression testing with screenshot comparison
- 15-minute max run time (parallelized)

**API E2E:**
- Full API workflow tests (create source → ingest event → check profile → build audience → sync destination)
- Run against staging after deployment
- Smoke tests also run against production (read-only) after deploy

### Chaos Engineering

Chaos engineering verifies the system handles failures gracefully:

**Experiments:**

<ComparisonTable
  headers={["Experiment", "What It Tests", "Expected Behavior"]}
  rows={[
    ["Kill random pod", "Pod recovery, HPA scaling", "Service recovers within 30s"],
    ["Network partition", "Cross-service resilience", "Graceful degradation, no data loss"],
    ["Kafka broker down", "Message queue resilience", "Consumer lag, auto-recovery"],
    ["BigTable latency spike", "Data layer resilience", "Circuit breaker trips, cached responses"],
    ["Memory pressure", "OOM handling", "Pod restarts, no cascading failure"]
  ]}
  highlightColumn={2}
/>

**Principles:**
- Start with hypothesis: "If X fails, the system should Y"
- Run in staging first, then controlled production experiments
- Always have a kill switch to stop the experiment
- Progressively increase blast radius

### Test Data Management

- **Factories** — Code-generated test data matching production schemas
- **Anonymized snapshots** — Production data with PII stripped for realistic testing
- **Cleanup** — Test data automatically purged after test runs
- **Isolation** — Each test run uses isolated org/namespace to prevent interference

<Callout type="important" title="Testing Culture">
Every PR must include tests. "I'll add tests later" is not acceptable. If a bug is found in production, the fix must include a test that would have caught it. Test coverage is tracked per service and reported in code review.
</Callout>
