### Testing Pyramid

Zeotap follows the testing pyramid model: a broad base of fast unit tests, a middle layer of integration tests, and a narrow top of slower but higher-fidelity E2E and chaos tests. Each layer catches different categories of bugs at different costs.

Unit tests form the foundation. They are fast, deterministic, and test individual components in isolation with all external dependencies mocked.

**Unit Testing Best Practices:**

- **Test behavior, not implementation** -- Assert on outputs and side effects, not internal method calls
- **Use dependency injection** -- Constructor injection makes classes testable without reflection hacks
- **Arrange-Act-Assert pattern** -- Clear separation of setup, execution, and verification
- **Descriptive test names** -- `shouldReturnErrorWhenAudienceExceedsMaxSize` not `testAudience3`
- **No flaky tests** -- Zero tolerance. Flaky tests are quarantined immediately and fixed within 48 hours
- **Test edge cases** -- Null inputs, empty collections, boundary values, concurrent access

<Expandable title="Java Unit Test Example" defaultOpen={false}>

```java
@ExtendWith(MockitoExtension.class)
class AudienceServiceTest {

    @Mock private AudienceRepository audienceRepository;
    @Mock private ProfileStore profileStore;
    @Mock private EventPublisher eventPublisher;

    @InjectMocks private AudienceService audienceService;

    @Test
    void shouldCreateAudienceWithValidRules() {
        // Arrange
        var orgId = "org_123";
        var request = AudienceRequest.builder()
            .name("High Value Users")
            .rules(List.of(Rule.greaterThan("ltv", 1000)))
            .build();

        when(audienceRepository.countByOrgId(orgId)).thenReturn(5L);

        // Act
        var result = audienceService.createAudience(orgId, request);

        // Assert
        assertThat(result.getName()).isEqualTo("High Value Users");
        assertThat(result.getStatus()).isEqualTo(AudienceStatus.EVALUATING);
        verify(eventPublisher).publish(any(AudienceCreatedEvent.class));
    }

    @Test
    void shouldRejectAudienceWhenOrgLimitExceeded() {
        // Arrange
        var orgId = "org_123";
        when(audienceRepository.countByOrgId(orgId)).thenReturn(500L);

        // Act & Assert
        assertThatThrownBy(() -> audienceService.createAudience(orgId, anyRequest()))
            .isInstanceOf(QuotaExceededException.class)
            .hasMessageContaining("Maximum audience limit");
    }
}
```

</Expandable>

### Integration Testing

Integration tests verify that a service works correctly with its real dependencies, catching issues that unit tests miss: serialization bugs, query syntax errors, connection pool behavior, and message format mismatches.

**Integration Test Categories:**

| Category | What It Tests | Infrastructure |
|---|---|---|
| Database integration | Query correctness, schema compatibility, transaction behavior | BigTable emulator, test BigQuery dataset |
| Kafka integration | Message serialization, consumer offset handling, error topics | Testcontainers Kafka |
| Cache integration | Cache hit/miss behavior, TTL, eviction, serialization | Testcontainers Redis |
| HTTP client integration | Upstream API contract, timeout handling, retry logic | WireMock |
| End-to-end service | Full request lifecycle within a single service | All of the above combined |

<Callout type="tip" title="Testcontainers Best Practice">
Use `@Testcontainers` with `@Container` annotations in JUnit 5 for automatic lifecycle management. Containers start before tests and stop after. Use `reuse = true` during local development to avoid container startup overhead on every test run.
</Callout>

### Contract Testing (Pact)

In a microservices architecture, the API contract between services is a critical reliability boundary. Pact ensures that consumers and providers agree on the API shape before deployment.

<ArchDiagram
  title="Pact Contract Testing Flow"
  steps={[
    { label: "Consumer Test", detail: "Define expected request/response interactions", icon: "📋" },
    { label: "Pact File", detail: "Generated JSON contract from consumer test", icon: "📄" },
    { label: "Pact Broker", detail: "Central registry for versioned contracts", icon: "🏢" },
    { label: "Provider Verification", detail: "Provider replays interactions against real service", icon: "✅" },
    { label: "Can-I-Deploy", detail: "Pre-deploy check: are all contracts satisfied?", icon: "🚦" }
  ]}
/>

**How Pact Works at Zeotap:**

1. **Consumer team** writes a Pact test defining the API interactions their service depends on (request method, path, headers, body, expected response)
2. **Pact generates** a contract file (JSON) capturing all interactions
3. **Contract published** to the Pact Broker with the consumer's version tag
4. **Provider team's CI** runs provider verification: replays every interaction from the contract against the actual running service
5. **can-i-deploy** check runs before each deployment, querying the Pact Broker to confirm all contracts between the deploying service and its dependents are satisfied
6. **Breaking changes** are caught before they reach any environment

<Expandable title="Pact Consumer Test Example (Java)" defaultOpen={false}>

```java
@ExtendWith(PactConsumerTestExt.class)
@PactTestFor(providerName = "profile-service", port = "8080")
class ProfileServiceConsumerTest {

    @Pact(consumer = "audience-engine")
    public V4Pact getProfilePact(PactDslWithProvider builder) {
        return builder
            .given("profile exists for user_abc in org_123")
            .uponReceiving("a request for user profile")
              .path("/api/v1/profiles/user_abc")
              .header("X-Org-Id", "org_123")
              .method("GET")
            .willRespondWith()
              .status(200)
              .header("Content-Type", "application/json")
              .body(newJsonBody(body -> {
                  body.stringType("userId", "user_abc");
                  body.stringType("orgId", "org_123");
                  body.object("attributes", attrs -> {
                      attrs.numberType("ltv", 1500);
                      attrs.stringType("segment", "high_value");
                  });
              }).build())
            .toPact(V4Pact.class);
    }

    @Test
    @PactTestFor(pactMethod = "getProfilePact")
    void shouldGetUserProfile(MockServer mockServer) {
        var client = new ProfileServiceClient(mockServer.getUrl());
        var profile = client.getProfile("org_123", "user_abc");

        assertThat(profile.getUserId()).isEqualTo("user_abc");
        assertThat(profile.getAttributes()).containsKey("ltv");
    }
}
```

**Cypress Best Practices:**
- **Stable selectors** -- Use `data-testid` attributes, never CSS classes or DOM structure
- **API seeding** -- Set up test data via API calls in `before()`, not through the UI
- **No waits** -- Use Cypress retry-ability and assertions, never `cy.wait(5000)`
- **Parallelized** -- Tests run across 4 parallel workers, keeping total suite under 15 minutes
- **Visual regression** -- Screenshot comparison for critical UI components (audience builder, dashboard charts)

**API E2E Tests:**

Full workflow tests that exercise the complete data pipeline:

1. Create a source via API
2. Ingest test events through the Collect endpoint
3. Wait for identity resolution to complete
4. Query the profile store to verify profile creation
5. Create an audience with rules matching the test data
6. Trigger a destination sync and verify delivery

These tests run against staging after every deployment and as read-only smoke tests against production post-deploy.

<Expandable title="k6 Load Test Example" defaultOpen={false}>

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate } from 'k6/metrics';

const errorRate = new Rate('errors');

export const options = {
  stages: [
    { duration: '2m', target: 100 },   // Ramp up to 100 VUs
    { duration: '10m', target: 500 },   // Ramp to 500 VUs (peak load)
    { duration: '15m', target: 500 },   // Sustain peak for 15 min
    { duration: '5m', target: 1000 },   // Spike to 2x peak
    { duration: '3m', target: 0 },      // Ramp down
  ],
  thresholds: {
    http_req_duration: ['p(99)<500'],    // P99 latency < 500ms
    errors: ['rate<0.01'],                // Error rate < 1%
  },
};

export default function () {
  const res = http.get('https://staging-api.zeotap.com/v1/audiences', {
    headers: { 'Authorization': `Bearer ${__ENV.AUTH_TOKEN}` },
  });

  check(res, {
    'status is 200': (r) => r.status === 200,
    'response time < 500ms': (r) => r.timings.duration < 500,
  });

  errorRate.add(res.status >= 400);
  sleep(0.1);
}
```

</Expandable>

**Load Test Schedule:**
- **Weekly automated runs** against staging (baseline validation)
- **Pre-release** -- Full-scale load test before every major release
- **Post-infrastructure change** -- Targeted load test after node pool resizing, BigTable rebalancing, or Kafka partition changes
- **Results archived** in GCS and compared against previous runs to detect performance regressions

### Chaos Engineering

Chaos engineering proactively tests system resilience by injecting controlled failures and observing system behavior.

<ComparisonTable
  headers={["Experiment", "Failure Injected", "Expected Behavior", "Blast Radius"]}
  rows={[
    ["Pod termination", "Kill random pods in a service", "HPA scales up, requests rerouted within 30s", "Single service"],
    ["Network partition", "Drop traffic between services", "Circuit breaker trips, graceful degradation", "Service pair"],
    ["Kafka broker failure", "Kill 1 of 3 Kafka brokers", "Consumers rebalance, no message loss", "Kafka cluster"],
    ["BigTable latency injection", "Add 5s latency to BigTable reads", "Circuit breaker opens, serve from cache", "Data layer"],
    ["Memory pressure", "Stress test pod memory limits", "OOM kill + restart, no cascading failure", "Single pod"],
    ["DNS failure", "Block DNS resolution for external services", "Cached responses, retry with backoff", "External integrations"]
  ]}
  highlightColumn={2}
/>

**Chaos Engineering Principles:**

1. **Start with a hypothesis** -- "If we kill 2 of 5 audience-engine pods, the remaining 3 should handle traffic within SLO while HPA scales up"
2. **Measure steady state first** -- Establish baseline metrics before injecting failure
3. **Minimize blast radius** -- Start in staging, start small, progressively increase scope
4. **Always have a kill switch** -- Automated abort if metrics exceed safety thresholds
5. **Run during business hours** -- So the team is available if something goes wrong
6. **Document findings** -- Every experiment generates a report with observations and action items

<Callout type="warning" title="Production Chaos">
Chaos experiments in production require VP Engineering approval, a documented rollback plan, and an active on-call engineer monitoring dashboards throughout the experiment. Production experiments are scoped to a single service or pod and never run during peak traffic windows.
</Callout>

### Test Data Management

Reliable tests require reliable test data. Zeotap uses a structured approach to test data creation and cleanup.

- **Factories** -- Code-generated test data matching production schemas. Each language has a `TestDataFactory` class producing valid entities with randomized fields.
- **Anonymized snapshots** -- Production data with all PII stripped (hashed emails, randomized names, preserved distributions) for realistic integration and load testing.
- **Isolated test orgs** -- Each test run creates a dedicated org in the test environment, preventing interference between parallel test suites.
- **Automatic cleanup** -- Test data and test orgs are purged 24 hours after creation via a scheduled cleanup job.
- **Seed data** -- Reference datasets (industry taxonomies, geo data, test destinations) are version-controlled and deployed with each environment.

### Code Coverage Standards

Coverage is tracked per service and enforced as a merge gate in CI.

- **Minimum thresholds** -- 80% line coverage for backend services, 70% for frontend, 75% for Spark jobs
- **Coverage tool** -- JaCoCo (Java), go tool cover (Go), coverage.py (Python), Istanbul (TypeScript)
- **Trend tracking** -- Coverage percentage reported on every PR with delta from base branch
- **No gaming** -- Coverage without meaningful assertions is caught in code review. A test that executes code without asserting behavior does not count.
- **New code rule** -- All new code in a PR must have 90%+ coverage, even if the overall service is below threshold

<Callout type="important" title="Testing Culture">
Every PR must include tests. "I'll add tests later" is not acceptable and will be rejected in review. If a bug is found in production, the fix must include a regression test that would have caught it. Testing is not a phase -- it is part of writing code.
</Callout>
