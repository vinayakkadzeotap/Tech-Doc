## 👤 Profile Store

The Profile Store is the single source of truth for unified customer profiles. Every resolved identity gets a profile that aggregates attributes, computed fields, and event history. This module covers the data model, storage architecture, and APIs.

**BigTable (Hot Layer):**
- Primary store for real-time profile access
- Row key: `{orgId}#{ucid}`
- Column families: `identity`, `attributes`, `behaviors`, `computed`, `consent`, `segments`
- Read latency: <10ms p99
- Write throughput: millions of upserts per second

**BigQuery (Warm Layer):**
- Full profile data replicated for analytical queries
- Powers reports, audience insights, and ad-hoc analysis
- Updated via streaming inserts from BigTable change stream
- Used for complex queries that BigTable can't serve (JOINs, aggregations)

**Redis (Cache):**
- L2 cache for frequently accessed profiles (top 10% by access frequency)
- TTL: 15 minutes
- Reduces BigTable read load by ~40%

### Profile Merge on Identity Resolution

When the ID Graph merges two clusters:

1. **Merge event** published to `profile.merge` Kafka topic
2. **Profile Merge Service** reads both profiles
3. **Attribute precedence** rules applied:
   - Most recent value wins for most attributes
   - Configurable per attribute (e.g., "first value wins" for date_of_birth)
   - Array attributes are unioned (e.g., interests, product categories)
4. **Merged profile** written to BigTable under the winning UCID
5. **Old UCID** becomes a redirect pointer to the new UCID
6. **Downstream consumers** notified of the merge

### Profile API

**Read Profile:**
```
GET /v1/profiles/{ucid}
Authorization: Bearer {api_key}

Response: {
  "ucid": "abc-123",
  "attributes": { "name": "John", "country": "DE" },
  "behaviors": { "total_purchases": 12, "last_visit": "2024-01-15" },
  "segments": ["high_value", "active_users"]
}
```

**Search Profiles:**
```
GET /v1/profiles?email=user@example.com
```
- Resolves identifier to UCID via ID Graph, then returns profile
- Supports search by any identifier type

### Delta Lake for Batch Updates

Batch profile updates (CRM imports, computed attribute refreshes) use Delta Lake:

- **ACID transactions** — batch updates are atomic, no partial writes
- **Time travel** — query profile state at any point in time
- **Schema evolution** — add new attributes without migration
- **Merge operations** — upsert millions of profiles in a single transaction

<Callout type="info" title="Profile Scale">
The Profile Store manages billions of profiles across all customers. BigTable's auto-scaling handles the storage and throughput, while BigQuery handles the analytical workload. This separation of concerns allows each layer to optimize for its access pattern.
</Callout>
