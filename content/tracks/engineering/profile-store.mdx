## 👤 Profile Store

The Profile Store is the single source of truth for unified customer profiles. Every resolved identity gets a profile that aggregates attributes, computed fields, and event history. This module covers the data model, storage architecture, and APIs.

**BigTable (Hot Layer):**
- Primary store for real-time profile access
- Row key: `{orgId}#{ucid}`
- Column families: `identity`, `attributes`, `behaviors`, `computed`, `consent`, `segments`
- Read latency: <10ms p99
- Write throughput: millions of upserts per second

**BigQuery (Warm Layer):**
- Full profile data replicated for analytical queries
- Powers reports, audience insights, and ad-hoc analysis
- Updated via streaming inserts from BigTable change stream
- Used for complex queries that BigTable can't serve (JOINs, aggregations)

**Redis (Cache):**
- L2 cache for frequently accessed profiles (top 10% by access frequency)
- TTL: 15 minutes
- Reduces BigTable read load by ~40%

### Profile Merge on Identity Resolution

When the ID Graph merges two clusters:

1. **Merge event** published to `profile.merge` Kafka topic
2. **Profile Merge Service** reads both profiles
3. **Attribute precedence** rules applied:
   - Most recent value wins for most attributes
   - Configurable per attribute (e.g., "first value wins" for date_of_birth)
   - Array attributes are unioned (e.g., interests, product categories)
4. **Merged profile** written to BigTable under the winning UCID
5. **Old UCID** becomes a redirect pointer to the new UCID
6. **Downstream consumers** notified of the merge

### Customer-Facing Profile API

The Profile API supports HTTPS and mTLS access:

**Base URLs:**
- Standard: `https://api.zeotap.com`
- Enhanced security: `https://mtls-api.zeotap.com`
- Auth: API Key header (`apikey: <api_key>`)

**Rate Limits:** 10,000 requests/min, 200 req/sec burst, 1KB avg response

**Read (Search) API:**
```
POST https://api.zeotap.com/cdp/v1/users/_search
{ "orgId": 4521, "search": { "email": ["user@example.com"] }, "regions": ["eu"] }
```
- Max 5 immutable IDs per call
- Returns attributes, ID graph, consent, marketing prefs, AI attributes

**Write (Upsert) API:**
```
POST https://api.zeotap.com/cdp/v1/users
{ "orgId": 4521, "region": "eu", "ids": { "email": "user@example.com" }, "upsert": { "profile": { "age_range": "25-34" } } }
```
- Response: 204 No Content + `X-Zeotap-UCID` header
- Writes via Pub/Sub → Dataflow enrichment → Delta MERGE INTO

**Delete (GDPR Erasure) API:**
```
POST https://api.zeotap.com/cdp/v1/users/_delete
{ "orgId": 4521, "region": "eu", "ids": { "email": ["user@example.com"] } }
```
- Bulk delete up to 400 IDs per call
- Erasure SLA: 7-10 business days

### Delta Lake Storage Design

The Profile Store is built on **Delta Lake** — providing ACID transactions on top of Apache Spark:

- **ACID Transactions** — Concurrent Spark jobs write safely via optimistic concurrency control in `_delta_log`
- **Change Data Feed (CDF)** — `delta.enableChangeDataFeed = true` streams only changed rows downstream
- **Schema Evolution** — `mergeSchema = true` allows new attributes from partners without rewriting
- **Time Travel** — `VERSION AS OF {n}` queries historical snapshots for audits and reproducible audience builds

**Delta Table Path Convention:**
```
gs://<bucket>/profile-store/<org_id>.<region>/delta-table/
  part-*.snappy.parquet     (data files)
  _delta_log/               (transaction log)
  _change_data/             (CDC files for streaming consumers)
```

**Merge Strategy (MERGE INTO):**
- Upsert by UCID — matched profiles update with `COALESCE(source, target)` for each field
- Unmatched profiles inserted with `created_at = NOW()`
- Attribute merge follows priority: first-party app > Tier 1 data partner > Tier 2 partner > ML predictions
