## 🛡️ Auth & IAM

### SSO Integration (SAML & OIDC)

Enterprise customers connect their corporate identity provider to Zeotap via SAML 2.0 or OpenID Connect, configured as Auth0 Enterprise Connections. Each customer organization maps to an Auth0 Organization entity.

<Expandable title="SAML Configuration Walkthrough" defaultOpen={false}>

When onboarding an enterprise customer with SAML SSO:

1. **Customer provides** their IdP metadata XML containing entity ID, SSO URL, and X.509 signing certificate
2. **Zeotap ops creates** an Auth0 Enterprise Connection of type SAMLP, uploading the metadata
3. **Auth0 generates** Service Provider metadata (ACS URL, Entity ID, signing cert) shared back to customer
4. **Customer configures** attribute mappings in their IdP: email, firstName, lastName, groups
5. **Auth0 connection** is linked to the customer's Auth0 Organization
6. **Domain routing** is configured so `@customer.com` emails are automatically directed to the SAML connection
7. **Testing** validates end-to-end flow with a test user before enabling for all users

The SAML assertion is consumed and validated by Auth0, which normalizes claims into a standard profile and issues a JWT. Backend services never interact with SAML XML directly -- they only see the normalized JWT.

</Expandable>

<Expandable title="OIDC Federation Setup" defaultOpen={false}>

For customers using Azure AD or Google Workspace:

1. **Customer registers** Zeotap as an OAuth application in their IdP, receiving a client ID and secret
2. **Zeotap configures** an Auth0 OIDC Enterprise Connection with the customer's discovery URL
3. **Scopes requested:** `openid profile email groups` for identity and group membership
4. **Group-to-role mapping** translates IdP groups (e.g., `zeotap-admins`) to Zeotap RBAC roles
5. **Token exchange:** Auth0 receives the id_token from the customer's IdP, maps claims, and issues its own JWT

OIDC is preferred over SAML for new integrations due to simpler configuration and native JSON token format.

</Expandable>

### JWT Token Flow

All authenticated requests carry a JWT access token issued by Auth0. The token lifecycle follows a strict pattern balancing security with user experience.

**Access Token Structure:**

```json
{
  "sub": "auth0|user_abc123",
  "aud": "https://api.zeotap.com",
  "iss": "https://zeotap.auth0.com/",
  "exp": 1710000900,
  "iat": 1710000000,
  "azp": "zeotap-dashboard-spa",
  "org_id": "org_enterprise_xyz",
  "permissions": ["read:audiences", "write:audiences", "read:profiles", "read:reports"],
  "roles": ["analyst"],
  "https://zeotap.com/tenant_id": "tenant_12345"
}
```

**Token Configuration:**

| Parameter | Value | Rationale |
|---|---|---|
| Access token TTL | 15 minutes | Short-lived to limit blast radius of token theft |
| Refresh token TTL | 24 hours | Absolute expiry, forces daily re-authentication |
| Refresh token rotation | Enabled | Each use issues a new refresh token, invalidates the old |
| Audience | `https://api.zeotap.com` | Tokens are scoped to the Zeotap API only |
| Algorithm | RS256 | Asymmetric signing for distributed verification |

**Token Validation Pipeline:**

1. API Gateway extracts `Authorization: Bearer {token}` header
2. Signature verified against Auth0 JWKS endpoint (keys cached with 24h TTL)
3. `exp` claim checked -- reject expired tokens
4. `aud` claim must match `https://api.zeotap.com`
5. `iss` claim must match `https://zeotap.auth0.com/`
6. `org_id` extracted and set as tenant context for the request
7. `permissions` array checked against endpoint requirements

<Callout type="warning" title="Token Validation is Non-Negotiable">
Every backend service MUST validate the JWT signature using Auth0's JWKS endpoint. The API gateway performs initial validation, but services must re-validate for sensitive operations. Never trust claims without signature verification. Never decode a JWT without validating it first.
</Callout>

**Silent Token Renewal:**

The Angular frontend uses Auth0's SDK to perform silent renewal before access token expiry. An invisible iframe requests a new token using the refresh token. If renewal fails (expired refresh token, revoked session), the user is redirected to the login page.

### RBAC Model

Zeotap implements hierarchical Role-Based Access Control operating at the organization level. Roles aggregate permissions, and users are assigned roles within the context of their organization.

    ["Org Admin", "Single org", "User management, source/destination config, all data operations", "Customer IT admin"],
    ["Analyst", "Single org", "Audience CRUD, profile queries, reporting, dashboard access", "Marketing analyst"],
    ["Developer", "Single org", "API key management, source config, schema management, debug tools", "Customer engineer"],
    ["Viewer", "Single org", "Read-only access to dashboards, audiences, and reports", "Business stakeholder"],
    ["Custom", "Single org", "Customer-defined permission set", "Varies by customer needs"]
  ]}
  highlightColumn={0}
/>

**Permission Granularity:**

Permissions follow a `resource:action` pattern providing fine-grained control:

- **Resources:** `audiences`, `profiles`, `sources`, `destinations`, `reports`, `settings`, `users`, `api_keys`
- **Actions:** `read`, `write`, `delete`, `admin`
- **Examples:** `audiences:write`, `destinations:delete`, `settings:admin`, `users:read`

**Enforcement Layers:**

Permission checks happen at two layers for defense in depth:

1. **API Gateway** -- Validates that the JWT `permissions` array contains the required permission for the endpoint. Unauthorized requests are rejected with `403 Forbidden` before reaching the service.

2. **Service Layer** -- Spring Security `@PreAuthorize` annotations enforce fine-grained access within business logic, including org-level data scoping:

```java
@PreAuthorize("hasPermission(#orgId, 'AUDIENCE', 'WRITE')")
public AudienceResponse createAudience(String orgId, AudienceRequest request) {
    // Tenant-scoped: only data belonging to orgId is accessible
}
```

<Expandable title="API Key Lifecycle" defaultOpen={false}>

API keys are managed through the Zeotap dashboard and follow a secure lifecycle:

1. User with `manage:api_keys` permission generates a key via the dashboard
2. The raw key is displayed **once** and never stored -- only a SHA-256 hash is persisted in the database
3. Each key is scoped to an organization and carries a fixed permission set defined at creation
4. Incoming requests with `X-API-Key` header are validated by computing the hash and performing a lookup
5. Keys support optional expiry dates and can be revoked immediately by any Org Admin
6. Rate limiting is applied per API key (configurable per org, default 1000 req/min)
7. All API key usage is logged for audit, including the key fingerprint, endpoint, and response status

API keys provide a simpler integration path but do not support the full RBAC model. For programmatic access with full role-based permissions, use OAuth Client Credentials instead.

</Expandable>

### Multi-Tenancy Isolation

Every request is scoped to an organization. The auth layer establishes tenant context, and every downstream component enforces isolation.

    { label: "Cache Layer", detail: "Redis key namespace: {tenant_id}:{resource}:{id}", icon: "⚡" }

/>

<Callout type="important" title="Tenant Context Propagation">
The `org_id` from the JWT is extracted at the API gateway and propagated through all downstream service calls via gRPC metadata or HTTP headers (`X-Tenant-Id`). Every database query, cache lookup, and Kafka message is scoped to this tenant context. Cross-tenant data access is architecturally impossible for non-super-admin roles.
</Callout>

**Implementation Details:**

- **Request-scoped context** -- A `TenantContext` object is set in a ThreadLocal (Java) or context value (Go) at the start of each request and cleared on completion
- **JPA/Hibernate filters** -- Global `@Filter` annotations automatically append `WHERE org_id = :tenantId` to all database queries
- **BigTable row key design** -- Row keys use the pattern `{org_id}#{entity_type}#{entity_id}`, ensuring range scans are inherently tenant-isolated
- **BigQuery authorized views** -- Each org accesses data through authorized views that enforce row-level security
- **Redis key namespacing** -- All cache keys follow `{tenant_id}:{service}:{key}` convention, preventing cross-tenant cache pollution

### Session Management

The Angular frontend manages sessions using Auth0's SPA SDK with silent token renewal and strict security controls.

- **Session duration:** Controlled by Auth0 SSO session settings (configurable per org, default 72 hours inactivity timeout)
- **Token renewal:** Silent iframe-based renewal initiated 60 seconds before access token expiry
- **Logout:** Triggers local session clear, Auth0 session termination, and optional federated IdP logout
- **Concurrent sessions:** Users may have multiple active sessions across devices (max 5, configurable per org)
- **Force logout:** Org Admins can revoke all refresh tokens for a user via the dashboard, terminating all active sessions
- **Audit logging:** All authentication events (login, logout, MFA challenge, token refresh, failed attempt) are logged to Cloud Logging with user ID, IP, user agent, and org context

<Callout type="tip" title="Security Headers">
All API responses include security headers: `Strict-Transport-Security` (HSTS with 1-year max-age), `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, and a strict `Content-Security-Policy`. CORS is restricted to `*.zeotap.com` domains only.
</Callout>
