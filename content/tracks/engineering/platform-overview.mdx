## 🏗️ Platform Architecture Overview

Zeotap CDP is an enterprise Customer Data Platform that unifies identity, enriches profiles, and activates audiences across channels — built on a microservices architecture deployed on Google Cloud Platform.

### What is Zeotap CDP?

Zeotap is a privacy-first Customer Data Platform that enables enterprises to collect first-party data, resolve identities across devices and channels, build rich customer audiences, and activate those audiences across 100+ destinations in real time.

The platform sits at the intersection of **data engineering**, **identity resolution**, **machine learning**, and **real-time activation**, serving telcos, retailers, financial services, and media companies globally.

**Collect** — Ingest first-party data from web, mobile, CRM, offline, and third-party sources via SDKs and batch pipelines.

**Unify** — Stitch identities across devices and channels into a single customer profile using the ID Graph engine.

**Activate** — Deliver audiences to DSPs, CRMs, social platforms, and custom destinations in real time or batch.

### System Architecture

Zeotap is built as a collection of domain-oriented microservices organized around the **Collect → Unify → Activate** data lifecycle. Services communicate via REST, gRPC, and event streaming (Kafka / GCP Pub/Sub).

### End-to-End Data Flow

The complete journey of a data event from collection to activation:

1. **Browser/App** sends event via SDK → `POST /collect/event`
2. **API Gateway** authenticates request and enforces rate limits
3. **Collect Service** validates schema, applies consent rules
4. **Kafka** receives the event on `events.raw.{orgId}` topic
5. **ID Graph** consumes event, resolves UCID, stitches device IDs
6. **Profile Store** upserts profile with new attributes
7. **Audience Engine** re-evaluates segment membership rules
8. **Destinations** push updated audience membership to activated platforms

<Callout type="info" title="Event Processing Latency">
The full pipeline from event collection to audience membership update is designed for **&lt;500ms p99 latency** for real-time use cases. Batch ingestion operates on a separate pipeline with configurable scheduling.
</Callout>
