### Continuous Deployment (ArgoCD)

**ArgoCD** monitors a GitOps config repository:

1. CI pipeline updates image tag in Helm values file
2. ArgoCD detects change in config repo
3. ArgoCD compares desired state (Git) with actual state (cluster)
4. ArgoCD applies the diff to the cluster
5. Health checks verify the deployment succeeded

**Key ArgoCD Features:**
- **Auto-sync** — Automatically deploy when config changes (staging)
- **Manual sync** — Require approval for production deployments
- **Rollback** — One-click rollback to any previous Git commit
- **Diff view** — See exactly what will change before deploying
- **Health monitoring** — Track deployment health and pod status

### Helm Charts

Each service is packaged as a Helm chart:

```
service-chart/
├── Chart.yaml          # Chart metadata, version
├── values.yaml         # Default configuration
├── values-dev.yaml     # Dev overrides
├── values-staging.yaml # Staging overrides
├── values-prod.yaml    # Production overrides
└── templates/
    ├── deployment.yaml
    ├── service.yaml
    ├── hpa.yaml
    ├── configmap.yaml
    └── ingress.yaml
```

### Environment Promotion

```
Feature Branch → Dev → Staging → Production
```

- **Dev:** Auto-deploy on every merge to main
- **Staging:** Auto-deploy, runs full integration test suite
- **Production:** Manual approval required, deployed during low-traffic windows
- **Hotfix:** Can bypass staging with VP Engineering approval

### Rollback Procedures

1. **Automatic:** If health checks fail after deploy, ArgoCD auto-rollbacks
2. **Manual:** Run `argocd app rollback {app} {revision}` or click in UI
3. **Git revert:** Revert the config repo commit, ArgoCD syncs
4. **Emergency:** Scale to 0 and redeploy known-good version

<Callout type="warning" title="Never Skip Staging">
Production incidents caused by untested changes are preventable. Always deploy to staging first, run smoke tests, and verify metrics before promoting to production. The only exception is a hotfix for an active P1 incident.
</Callout>
