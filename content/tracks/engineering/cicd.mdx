## 🔄 CI/CD Pipelines

Zeotap follows a GitOps workflow where the desired state of every environment is declared in Git. This module covers the complete pipeline from code commit to production deployment, including CloudBuild for CI, ArgoCD for CD, Helm packaging, deployment strategies, feature flags, and rollback procedures.

### GitOps Workflow

In the GitOps model, Git is the single source of truth for both application code and infrastructure configuration. No manual `kubectl apply` commands, no ad-hoc configuration changes. Everything flows through Git.

**Two-Repository Model:**

Zeotap separates application code from deployment configuration:

| Repository | Contains | Trigger |
|---|---|---|
| Application repo (e.g., `zeocore-api`) | Source code, Dockerfile, unit tests, CloudBuild config | PR creation and merge |
| Config repo (`k8s-deployments`) | Helm value files per service per environment | Image tag update from CI |

This separation ensures that deployment configuration changes are versioned, reviewed, and auditable independently from code changes.

### Continuous Integration (CloudBuild)

CloudBuild runs the CI pipeline on every PR and merge to main. The pipeline is defined as a `cloudbuild.yaml` in each application repository.

    ["Push Image", "Tag and push to Artifact Registry", "Google Artifact Registry", "Block deploy"],
    ["Update Config", "Patch Helm values with new image tag", "yq, git commit to config repo", "Block deploy"]
  ]}

<Expandable title="CloudBuild Configuration Example" defaultOpen={false}>

A typical `cloudbuild.yaml` for a Java Spring Boot service:

```yaml
steps:
  - id: 'lint'
    name: 'gradle:8-jdk17'
    args: ['gradle', 'checkstyleMain', 'checkstyleTest']

  - id: 'test'
    name: 'gradle:8-jdk17'
    args: ['gradle', 'test', 'jacocoTestReport']
    waitFor: ['lint']

  - id: 'security-scan'
    name: 'snyk/snyk:gradle'
    args: ['snyk', 'test', '--severity-threshold=high']
    waitFor: ['-']  # Run in parallel with lint

  - id: 'build-image'
    name: 'gradle:8-jdk17'
    args: ['gradle', 'jib', '--image=${_REGISTRY}/${_SERVICE}:${SHORT_SHA}']
    waitFor: ['test', 'security-scan']

  - id: 'container-scan'
    name: 'aquasec/trivy'
    args: ['image', '--exit-code', '1', '--severity', 'CRITICAL',
           '${_REGISTRY}/${_SERVICE}:${SHORT_SHA}']
    waitFor: ['build-image']

  - id: 'update-config'
    name: 'mikefarah/yq'
    entrypoint: 'bash'
    args:
      - '-c'
      - |
        git clone https://github.com/zeotap/k8s-deployments.git
        cd k8s-deployments
        yq e '.image.tag = "${SHORT_SHA}"' -i services/${_SERVICE}/values-dev.yaml
        git commit -am "chore: update ${_SERVICE} to ${SHORT_SHA}"
        git push
    waitFor: ['container-scan']

substitutions:
  _REGISTRY: 'europe-west3-docker.pkg.dev/zeotap-prod/services'
  _SERVICE: 'zeocore-api'

options:
  machineType: 'E2_HIGHCPU_8'
```

Key features:
- **Parallel execution** of independent stages (lint + security scan)
- **Jib** for Java image builds (no Docker daemon required, layer caching)
- **Trivy** for container vulnerability scanning
- **Automated config update** commits the new image tag to the GitOps repo

</Expandable>

**Build Performance:**
- Average CI pipeline duration: 6-8 minutes for Java services, 3-4 minutes for Go services
- Docker layer caching via Artifact Registry for faster image builds
- Gradle/Go build caches stored in GCS for incremental compilation
- Parallel stage execution reduces critical path
- Build notifications to Slack with status, duration, and links

### Continuous Deployment (ArgoCD)

ArgoCD watches the `k8s-deployments` config repo and continuously reconciles the desired state (Git) with the actual state (GKE cluster). It is the only mechanism for deploying to Kubernetes.

**Deployment Flow:**

1. CI pipeline updates image tag in `k8s-deployments/services/{service}/values-{env}.yaml`
2. ArgoCD detects the Git commit within 3 minutes (polling interval) or immediately via webhook
3. ArgoCD renders the Helm chart with updated values and computes the diff
4. For dev/staging: auto-sync applies the diff immediately
5. For production: engineer reviews the diff in ArgoCD UI and clicks "Sync"
6. ArgoCD applies resources in sync-wave order (ConfigMaps first, then Deployments, then Services)
7. Health checks verify all pods are running and readiness probes pass
8. ArgoCD marks the sync as "Healthy" or "Degraded"

<Callout type="info" title="Self-Heal Mode">
ArgoCD's self-heal mode automatically reverts any manual changes made directly to the cluster (e.g., `kubectl edit`). This enforces the GitOps principle that Git is the sole source of truth. If something needs to change, it goes through a PR to the config repo.
</Callout>

### Helm Charts

Every microservice is packaged as a Helm chart with environment-specific value overrides. A shared base chart provides consistent defaults across all services.

```
k8s-deployments/
├── charts/
│   └── zeotap-service/          # Shared base chart
│       ├── Chart.yaml
│       ├── values.yaml           # Defaults (replicas, resources, probes)
│       └── templates/
│           ├── deployment.yaml
│           ├── service.yaml
│           ├── hpa.yaml
│           ├── pdb.yaml
│           ├── configmap.yaml
│           ├── serviceaccount.yaml
│           └── servicemonitor.yaml
├── services/
│   ├── zeocore-api/
│   │   ├── values-dev.yaml
│   │   ├── values-staging.yaml
│   │   └── values-prod.yaml
│   ├── id-graph-service/
│   │   ├── values-dev.yaml
│   │   ├── values-staging.yaml
│   │   └── values-prod.yaml
│   └── audience-engine/
│       └── ...
└── platform/                     # Platform services (ArgoCD, Prometheus, Istio)
```

**Value Override Hierarchy:**

Base chart `values.yaml` provides sane defaults (2 replicas, 512Mi memory, standard probes). Environment-specific files override only what differs (image tag, replica count, resource limits, feature flags, external endpoints).

### Deployment Strategies

Different deployment strategies are used depending on the risk profile and service type.

<ComparisonTable
  headers={["Strategy", "Mechanism", "When to Use", "Rollback Speed"]}
  rows={[
    ["Rolling Update", "Gradually replace old pods; maxSurge=25%, maxUnavailable=0", "Default for all services", "Automatic (failed readiness)"],
    ["Canary", "Route 5% traffic to new version via Istio, monitor error rate", "High-risk changes, new features", "Instant (shift traffic back)"],
    ["Blue-Green", "Deploy new version alongside old, atomic traffic switch", "Database migrations, breaking changes", "Instant (switch traffic back)"],
    ["Feature Flag", "Deploy code to all pods, enable via LaunchDarkly flag", "Experimental features, gradual rollout", "Instant (toggle flag off)"]
  ]}
  highlightColumn={0}
/>

<Expandable title="Canary Deployment Process" defaultOpen={false}>

For high-risk deployments, the canary process provides safe incremental rollout:

1. **Deploy canary** -- New version deployed as a separate Deployment with 1 replica
2. **Traffic split** -- Istio VirtualService routes 5% of traffic to the canary
3. **Monitor (15 min)** -- Watch error rate, latency P99, and business metrics on canary-specific dashboard
4. **Increase to 25%** -- If metrics are healthy, increase canary traffic share
5. **Monitor (15 min)** -- Second observation window at higher traffic
6. **Promote to 100%** -- Scale up canary, scale down old version
7. **Cleanup** -- Remove old Deployment after 1 hour (quick rollback window)

**Automated canary analysis** is under development using Flagger, which will automatically promote or rollback based on Prometheus metrics.

</Expandable>

### Feature Flags

Feature flags decouple deployment from release, allowing code to be deployed to production but activated only when ready.

<TechStack
  title="Feature Flag Stack"
  items={[

    { name: "Server-side SDK", role: "Java, Go, Python SDKs for backend evaluation" },
    { name: "Client-side SDK", role: "Angular SDK for frontend flag evaluation" },
    { name: "Targeting Rules", role: "Per-org, per-user, percentage-based rollout" },
    { name: "Kill Switch", role: "Instant disable for any feature in production" }
  ]}
/>

**Flag Lifecycle:**
1. Engineer creates flag in LaunchDarkly with naming convention: `enable-{feature}-{team}`
2. Code checks flag at runtime, defaults to disabled
3. Flag enabled in dev/staging for testing
4. Production rollout: enable for internal org first, then 10% of customers, then 50%, then 100%
5. After full rollout and stability period (2 weeks), flag is removed from code (tech debt cleanup)

<Callout type="warning" title="Flag Hygiene">
Feature flags are technical debt. Every flag has an assigned owner and a removal date. Flags older than 90 days without a removal plan are flagged in weekly engineering review. Stale flags increase code complexity and testing burden.
</Callout>

### Rollback Procedures

Rollbacks are a first-class operation, designed to be fast and reliable.

**Rollback Methods (ordered by speed):**

1. **ArgoCD auto-rollback** -- If health checks fail within 5 minutes of sync, ArgoCD automatically reverts to the previous healthy state. No human intervention required.

2. **ArgoCD manual rollback** -- Click "History" in ArgoCD UI, select the target revision, click "Rollback." Takes effect within 30 seconds. This is the standard rollback method.

3. **Git revert** -- Revert the config repo commit that updated the image tag. ArgoCD detects the revert and syncs. Takes 3-5 minutes. Creates a clear audit trail.

4. **Feature flag kill switch** -- If the issue is in a flagged feature, disable the flag in LaunchDarkly. Takes effect within seconds. No deployment change required.

5. **Emergency scale-to-zero** -- For catastrophic issues, scale the service to 0 replicas, investigate, and redeploy a known-good version. Last resort only.

<Callout type="important" title="Rollback Validation">
After any rollback, verify recovery by checking: (1) error rate returns to baseline on Grafana, (2) no Kafka consumer lag building up, (3) no customer-facing errors on Statuspage. Log the rollback in the incident channel with the reason, affected version, and rolled-back-to version.
</Callout>
