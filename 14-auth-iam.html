IAM Architecture

Zeotap's authentication and authorisation layer uses a two-tier model:

  • Authentication — Handled by Firebase Authentication (Google Identity Platform), providing SSO via Google, SAML, and email/password.
  • Authorisation — Handled by Akbar (akbar-the-authenticator), Zeotap's internal RBAC library. Akbar validates JWT tokens, resolves user roles, and enforces product-level permissions.

Akbar RBAC Authenticator

The akbar-the-authenticator Maven library (com.zeotap.backend:rbac-authenticator) is a shared Java library embedded in all Zeotap backend services. It provides:

  • JWT validation — Verifies Zeotap-issued session tokens using shared secret or RSA public key
  • RBAC resolution — Maps user+org to product entitlements and permission sets
  • Async validation — Non-blocking token check via AsyncRbacAuthUtil
  • Exception handling — InvalidTokenException, RbacAuthException, InvalidResponseException

JOLT Token Transformation

Akbar uses the JOLT library (com.bazaarvoice.jolt) to transform token payloads between Firebase claim format and Zeotap internal RBAC format without manual field mapping:

// JOLT spec: Firebase claims → Zeotap RBAC token
[
  {
    "operation": "shift",
    "spec": {
      "uid":      "userId",
      "email":    "email",
      "org_id":   "orgId",
      "custom_claims": {
        "products": "products",
        "roles":    "roles"
      }
    }
  },
  {
    "operation": "default",
    "spec": {
      "tokenVersion": "2",
      "issuer": "zeotap"
    }
  }
]

Firebase Authentication

The firebase-authenticator library wraps the Firebase Admin SDK for server-side token verification. It is used by Akbar to validate incoming Firebase ID tokens before issuing Zeotap session JWTs.

// Server-side Firebase token verification (Java)
FirebaseApp.initializeApp(FirebaseOptions.builder()
    .setCredentials(GoogleCredentials.getApplicationDefault())
    .setProjectId("zeotap-prod")
    .build());

// Verify Firebase token
FirebaseToken decodedToken = FirebaseAuth.getInstance()
    .verifyIdToken(firebaseIdToken);

String uid    = decodedToken.getUid();
String email  = decodedToken.getEmail();
Map claims = decodedToken.getClaims();

ARBEC — Access Role-Based Entitlement Control

ARBEC is Zeotap's comprehensive access management system that controls user permissions across all products and services. It operates on top of the Akbar/Firebase layer to enforce what a user can do once authenticated — mapping users to product groups and granular feature permissions (~650 total features).

Access Hierarchy

ARBEC follows a strict four-level hierarchy from broad package access down to individual feature permissions:

LevelDescriptionExample
UsersIndividual users belonging to an organisationuser_id: 12345
Package RolesRoles that grant access to a product packageCDP Admin, Targeting User
Product GroupsPermission sets for a specific productSegments Read-Only, Journeys Full Access
FeaturesGranular permissions defining specific actions (~650 total)audiences.create, destinations.delete

Organisation Hierarchy

Organisations follow a parent-child tree with two key rules:

  • Parent caps child access — A parent org's product access is the maximum ceiling for all child orgs. A child cannot have more access than its parent.
  • Zeotap is global parent — Zeotap itself is the root organisation with access to all orgs in the system. Used for internal tooling and cross-org support operations.

Package–Product Mapping

Default product access per package is hardcoded in user_service/common.py:

PACKAGE_PRODUCT_MAP = {
    1: [1, 12, 21, 15, 23],  # CDP Package     — CDP, RB, Destinations, etc.
    2: [1, 5, 12],           # Targeting Package — Targeting products
    3: [1, 19]               # Other Package
}

Additional products (e.g., Journeys) can be toggled on/off per org via the UI. These are stored in the product_org_map database table and take effect immediately.

ARBEC Database Architecture

ARBEC data is stored in the Arc DB, accessible via Redash for support queries. The following tables are the core of the permission model:

TablePurpose
user_product_groupLinks users to product groups for specific orgs/products — the central join table
product_feature_group_mapMaps product groups → their individual feature permissions
product_org_mapStores organisation-product access mappings (enabled products per org)
product_groupDefines product groups, including custom roles (custom groups have non-null org_id)
group_role_mapMaps custom roles → product groups
product_featuresMaster list of all ~650 features available in the system
user_rolesExposes custom roles to clients for display in the Unity UI

Key SQL Queries (Support Cheat Sheet)

Use these on Redash (Arc DB) to debug access issues:

-- Check all product groups a user belongs to in an org
SELECT * FROM user_product_group
WHERE user_id = {user_id}
AND org_id = {org_id};

-- Check which features a product group grants
SELECT f.* FROM product_features f
JOIN product_feature_group_map m ON f.id = m.feature_id
WHERE m.product_group_id = {group_id};

-- Check which products an org has access to
SELECT * FROM product_org_map
WHERE org_id = {org_id};

Custom Roles & Folder Management

Custom Roles

Custom Roles allow organisations to create roles with bespoke permission sets, bypassing the predefined package roles. This addresses use cases requiring fine-grained control beyond CDP Admin / Targeting User defaults.

  • Implementation — Custom roles create new product_group rows with a non-null org_id, scoping them to that organisation.
  • Role mapping — Stored in group_role_map. Exposed to clients via user_roles.
  • Restriction — Certain permissions are reserved for CDP Admin and Zeotap Admin roles. Do not create custom roles for these — enabling them via UI toggles is not permitted.

Folder Management

Folder-level access restrictions (Audience folders, Journey folders) have been migrated from independent services into ARBEC, enabling centralised folder-level RBAC:

  • Folder access is configured at the Product Group level in the product_group table.
  • A null folder value grants access to all folders — the default open state.
  • Populating the folder field restricts the group to only the listed folder IDs.

Troubleshooting Guide

Issue 1: User Cannot Access a Product

User reports they can't access a product they should have access to. Follow these steps:

1
Check user_product_group — Verify the user has an entry linking them to the required product group for their org.
2
Check product_org_map — Confirm the user's organisation has the product enabled.
3
Check package role — Ensure the user's package role includes the product in PACKAGE_PRODUCT_MAP.

Issue 2: User Has Product Group But Missing a Feature

1
Check product_feature_group_map — Verify the product group is linked to the required feature ID.
2
Check product_features — Confirm the feature exists and is active in the master feature table.
3
Custom role check — If using a custom role, use the custom_roles API to verify the feature is bound directly.

ARBEC Internal APIs

Token Decrypt API

Validates a user's JWT and returns their permission payload for a given org and product. Used internally by all Zeotap services to check specific user permissions.

POST http://rbac-prod.zeotap.net/internal/api/v3/token/payload 
curl --location 'http://rbac-prod.zeotap.net/internal/api/v3/token/payload' \
  --header 'accept: application/json' \
  --header 'content-type: application/json' \
  --header 'authorization: {internal_token}' \
  --data '{
    "token": "{user_jwt_token}",
    "org_ids": [2110],
    "product_ids": [12]
  }'
ParameterTypeDescription
tokenstringUser's JWT session token
org_idsarray[int]Organisation IDs to check permissions for
product_idsarray[int]Product IDs to scope the permission check

Custom Roles API

Returns a custom role's direct feature bindings, bypassing the product group layer. Useful for debugging custom role permissions without querying the DB.

GET https://account-v3.zeotap.com/api/v5/organization/roles/?orgIds={orgId} 
curl 'https://account-v3.zeotap.com/api/v5/organization/roles/?orgIds=1894' \
  -H 'accept: application/json' \
  -H 'authorization: {user_token}' \
  -H 'content-type: application/json' \
  -H 'zeotap-org-id: 1894'
