Identity Resolution & ID Graph

Zeotap's identity layer stitches together multiple device identifiers, PII hashes, and behavioral signals into a single Universal Customer ID (UCID) — using a probabilistic + deterministic graph engine backed by Google BigTable and Redis.

id-graph-lib-v2-go (Go 1.24)
BigTable · Redis · Pub/Sub
PGP-encrypted PII

Overview

Identity resolution is the process of linking multiple partial identifiers (cookie ID, device ID, hashed email, phone number, etc.) to a single canonical customer record. Zeotap's ID graph performs this in near real-time as events arrive, enabling downstream systems to operate on a unified customer view.

ℹ️
Deterministic vs Probabilistic Matching

Zeotap uses deterministic matching (exact ID overlap) as the primary resolution method, supported by probabilistic matching (device fingerprinting, behavioral signals) for anonymous-to-known stitching across devices.

Universal Customer ID (UCID)

The UCID is Zeotap's canonical identifier assigned to every resolved customer entity. It is a deterministic, globally unique ID generated from the first known deterministic identifier and propagated through the ID graph as new IDs are linked.

go
// types/ucid.go
// UCID structure from id-graph-lib-v2-go

type UCID struct {
    ID          string    // Canonical UCID (UUID v4)
    OrgID       int64     // Organization scoping
    CreatedAt   time.Time
    LastSeenAt  time.Time
    Confidence  float64   // Resolution confidence [0.0 - 1.0]
    IDCount     int       // Number of linked identifiers
}

type UserProfile struct {
    UCID        UCID
    Identifiers []Identifier
    Attributes  map[string]interface{}
    Segments    []string
    Consent     ConsentRecord
    RawPII      RawPIIRecord   // PGP-encrypted
}

type Identifier struct {
    IDType      string    // e.g. "email_sha256", "gaid", "ucid"
    IDValue     string
    Source      string    // origin system
    Timestamp   time.Time
    IsPrimary   bool
}

Data Store Design

BigTable Schema

Google Cloud BigTable stores the identity graph as a wide-column table. Each row key encodes the organization ID and identifier value; column families partition identity vs profile data.

text — BigTable Schema
Table: zeotap-identity-graph

Row Key format:
  {org_id}#{id_type}#{id_value_hash}
  Example: "1234#email_sha256#a3b4c5d6..."

Column Families:

  ids:                    // Identity mapping
    ids:ucid               → "ucid_9f8e7d6c-..."
    ids:linked_ids         → JSON array of linked identifiers
    ids:id_count           → "15"
    ids:created_at         → "2024-01-15T10:30:00Z"
    ids:last_seen          → "2026-02-27T09:15:00Z"
    ids:confidence         → "0.97"

  profile:               // User attributes
    profile:age_band       → "25-34"
    profile:gender         → "M"
    profile:interests      → JSON array
    profile:segments       → JSON array of segment IDs

  consent:              // Consent state
    consent:analytics      → "1"
    consent:marketing      → "0"
    consent:updated_at     → "2026-01-20T14:00:00Z"

  pii_enc:              // PGP-encrypted raw PII (restricted)

Multi-Level Cache Architecture

To serve identity lookups at <5ms p99, the library implements a two-tier cache chain that intercepts reads before they hit BigTable.

🗄️ Cache Read Path
sequenceDiagram participant S as Service participant L1 as L1 Cache (Memory) participant L2 as L2 Cache (Redis) participant BT as BigTable S->>L1: GetUCID(orgId, idType, idValue) alt L1 Hit (TTL: 60s) L1-->>S: UCID (from memory) else L1 Miss L1->>L2: Forward lookup alt L2 Hit (TTL: 300s) L2-->>L1: UCID (populate L1) L1-->>S: UCID else L2 Miss L2->>BT: ReadRow(rowKey) BT-->>L2: Row data L2-->>L1: UCID (populate both caches) L1-->>S: UCID end end
go
// cache/chain.go — Cache chain implementation

type CacheChain struct {
    L1  *MemoryCache   // sync.Map — goroutine-safe
    L2  *RedisCache    // Redis with configurable TTL
    DB  IDRDatabase    // BigTable DAO interface
}

func (c *CacheChain) GetUCID(ctx context.Context,
    orgID int64, idType, idValue string) (string, error) {

    key := buildCacheKey(orgID, idType, idValue)

    // 1. L1 — in-process memory
    if ucid, ok := c.L1.Get(key); ok {
        return ucid, nil
    }

    // 2. L2 — Redis
    if ucid, err := c.L2.Get(ctx, key); err == nil {
        c.L1.Set(key, ucid, 60*time.Second)
        return ucid, nil
    }

    // 3. BigTable — source of truth
    profile, err := c.DB.GetProfile(ctx, orgID, idType, idValue)
    if err != nil {
        return "", err
    }

    // Backfill caches
    c.L2.Set(ctx, key, profile.UCID.ID, 5*time.Minute)
    c.L1.Set(key, profile.UCID.ID, 60*time.Second)
    return profile.UCID.ID, nil
}

Library API Reference

Core functions exposed by id-graph-lib-v2-go for integration by upstream services:

Function Signature Description
GetUCID (ctx, orgID, idType, idValue) → (ucid, error) Look up or resolve UCID for a given identifier. Creates new UCID if not found.
GetProfile (ctx, orgID, ucid) → (UserProfile, error) Fetch the full customer profile by UCID, including all linked IDs, attributes, segments.
UpsertIdentifier (ctx, orgID, ucid, identifier) → error Link a new identifier to an existing UCID. Triggers merge if identifier is already linked to another UCID.
MergeUCIDs (ctx, orgID, primary, secondary) → error Merge two UCIDs into one. All identifiers from secondary are re-linked to primary.
DeleteProfile (ctx, orgID, ucid) → error GDPR right-to-erasure. Deletes all rows related to a UCID across all identifier types.
🚨
GDPR Deletion — Irreversible

DeleteProfile performs a hard delete across BigTable, all cache layers, and triggers a Pub/Sub event for downstream deletion. This operation cannot be undone. Requires gdpr:delete IAM role. See Chapter 13 for the full GDPR workflow.
